# save_rule_doc — 保存规则文件到规则库

本技能随包下发一个纯标准库脚本 `scripts/smart_doc_add.py`，它把 sandbox 里的文件
multipart 上传到立项审查后端的 `POST /api/standard-docs` 完成入库。后端地址写在同目录的
`scripts/api_base.txt`（部署时由你替换成可达地址，见包内 README）。

## 何时触发
用户在聊天中**已用“+”上传了一个文件**（文件已在当前 session 的 sandbox 中），并表达把它
存为规则文件的意图，例如：
- “这是规则文件”“把这个存为规则文件”
- “加入规则库 / 保存到规则库”
- “这是一份规则文档，存一下”

## 前置判断
1. 确认 sandbox 中确有用户刚上传的文件，拿到它的**绝对路径**。
2. 判断它是否为**规则类文档**（政策、指南、规范、办法等）。
   - 若明显不是规则文档（随手的图片、无关表格等），**先说明并询问用户是否仍要保存**，不擅自入库。
   - 若用户意图不明确，先澄清再动作。

## 执行
对确认要保存的文件，用 Bash 工具运行下面的步骤（脚本路径随运行环境可能不同，故先定位）：

```bash
# 1) 定位随本技能下发的脚本
SCRIPT="$(find / -path '*save_rule_doc/scripts/smart_doc_add.py' 2>/dev/null | head -1)"
# 2) 读取后端地址（部署时已写入脚本同目录的 api_base.txt）；tr 去掉可能的 CR/LF
API="$(cat "$(dirname "$SCRIPT")/api_base.txt" | tr -d '\r\n')"
# 3) 上传入库（每个文件绝对路径作为一个参数，可多个）
SMART_DOC_API="$API" python3 "$SCRIPT" "<sandbox 中的文件绝对路径>"
```

> 若 `find` 没找到脚本，说明技能资源未正确同步，请如实告知用户“技能脚本缺失，无法入库”，不要伪造结果。

脚本会向 stdout 打印每个成功入库文件的 `doc_code=... title=... file=...`，向 stderr 打印失败项，
并用**退出码**表达整体结果。

## 解读结果（按退出码）
| 退出码 | 含义 | 你应如何回报 |
|---|---|---|
| 0 | 全部成功 | 报告每个文件的标题与编号，例：“已存入规则库：政策A（SD-abc123）” |
| 1 | 用法错误（没给文件参数） | 说明没有可保存的文件 |
| 2 | 文件不存在/不可读 | 说明没找到文件，请用户确认是否已上传 |
| 3 | 连不上规则库后端 | 说明后端暂时不可达（可能隧道未开/地址错），建议稍后重试 |
| 4 | 后端返回错误（非 2xx） | 说明入库失败（附 stderr 原因），不要声称成功 |
| 5 | 部分/全部文件失败 | 逐个报告：成功的报标题+编号，失败的报 stderr 里的原因（如超过大小上限） |
| 6 | 后端地址未配置 | 这是部署问题：`api_base.txt` 为空或未读到，提示需在包内填真实地址（占位符未替换通常表现为退出码 3） |

## 铁律
- **只有退出码 0 且 stdout 出现 `doc_code=...` 才能说“已保存成功”。** 其他情况一律如实说明失败原因，绝不谎报。
- 不要凭空编造 `doc_code` 或标题——只转述脚本的实际输出。
- 不得绕过本脚本直连数据库或自行拼 SQL；所有入库一律走该脚本（它内部调后端的带校验接口）。
