# 立项审查系统 — 用户权限管理 + 任务分发 改造方案

> 状态：分析稿（未改任何业务代码）
> 目标：新增「管理员端（任务创建 / 报告上传 1+4 / 受理分发 / 审查台账）」+「普通用户端（收到分发任务 → 看到自己的项目 → 进入 5 个审查按钮）」。
> 其中 5 个审查按钮（综合论证报告 / 经济性 / 技术体质 / 体系贡献率 / 通用质量特性）本方案只留占位入口，由你后续实现。

---

## 0. 现状结论（为什么这块基本是从零做）

| 能力 | 现状 |
|---|---|
| 登录 / 会话 | ❌ 完全没有。`auth.py` 只是可选共享密钥 `X-API-Key`，空值放行，仅护写/删端点 |
| 当前用户上下文 | ❌ 后端无 `get_current_user`，前端无用户 store；`created_by` 等只是审计列 |
| 角色 / 权限门控 | ❌ 表存在（`role/user_role/permission/role_permission`）但无任何代码使用 |
| 用户密码 | ❌ `sys_user` 无 `password_hash` 列；种子里只有 1 个 `__system__` 用户、无角色 |
| 任务归属（分发给某人） | ❌ 无任何字段把一个审查任务指派给某个普通用户 |
| 前端路由 | 自研 zustand `useRouteStore` + `App.tsx` switch，**非 react-router**；加页面要三处联动 |

结论：**要把「可选密钥鉴权」升级成「真正的登录 + RBAC + 任务归属」，并新增一条"任务→报告→分发→我的任务"业务线。**

## 1. 领域建模：把需求映射到数据

### 1.1 概念对齐
- **任务(review_task)** = 一个待审查的立项项目。管理员创建。
- **报告(task_report)** = 任务下的 **1+4** 份报告，类型固定为 5 种，正好对应普通用户端的 5 个审查按钮：

  | report_type | 中文 | 对应审查按钮 |
  |---|---|---|
  | `comprehensive` | 综合论证报告 | 综合论证报告审查 |
  | `economy` | 经济性 | 经济性审查 |
  | `tech_system` | 技术体质 | 技术体质审查 |
  | `system_contribution` | 体系贡献率 | 体系贡献率审查 |
  | `general_quality` | 通用质量特性 | 通用质量特性审查 |

- **受理分发** = 把任务的 `assignee_id` 指派给某个普通用户，并记录时间/操作人；普通用户可"接受受理"。
- **审查台账** = 管理员对全部任务的列表视图（含分发状态、受理人、审查进度）。
- **我的任务** = `assignee_id == 当前用户` 的任务列表。

### 1.2 与现有表的取舍（**关键决策**）

现有已有一条重的业务线：`review_batch → declared_project → application_package → material_file → review_round/round_check`。它服务于 AI 审查流水线，且 `application_package`/`declared_project` 有一堆必填外键（project_type / declaring_unit / applicant_person / secrecy_level…）。

- **方案 A：复用现有表**（任务=application_package，报告=material_file）
  - 优点：天然接入现有审查流水线（round_check 等），5 个审查按钮能复用既有审查执行。
  - 缺点：创建一个任务要先造 declared_project + 一堆主数据外键，管理员端表单很重；且现有 material_file 无"5 类报告"语义。
- **方案 B：新增轻量任务层**（新表 `review_task` + `task_report`，自带 `assignee_id`）✅ **推荐**
  - 优点：与新 UX 一一对应，字段最少，耦合最低，落地最快、风险最小。
  - 缺点：与现有申报包模型并行；将来 5 个审查按钮若要接入 round_check，需要一层映射（`review_task.package_id` 预留即可）。

> **推荐 B**：先把"权限 + 任务分发 + 我的任务"这条线独立跑通，`review_task` 上预留一个可空 `package_id`，将来做审查按钮时再桥接到现有审查流水线，互不阻塞。下文按方案 B 展开（方案 A 差异见 §6）。

## 2. 数据库改动（方案 B）

> 建表权威脚本：`docs/file/审查系统-schema_mysql.sql`；`db.py` 不做 `create_all`，所以改表需**手写迁移 SQL 并同步 `models.py`**。建议新增 `docs/file/migrations/2026-07-权限与任务.sql`（幂等）。

### 2.1 复用/补齐 5A 用户权限域
```sql
-- 1) 给 sys_user 加密码列（现无）
ALTER TABLE sys_user ADD COLUMN password_hash VARCHAR(255) NULL AFTER display_name;

-- 2) 角色种子（表已存在）
INSERT INTO role (code, name) VALUES
  ('admin','管理员'), ('reviewer','评审专家') ;   -- reviewer=普通用户

-- 3) 建一个管理员账号 + 绑定角色（密码由后端脚本写 hash，勿明文）
--    见 §3.5 seed 脚本
```
- 沿用现成表：`sys_user`（用户）、`role`/`user_role`（角色绑定）。
- `permission`/`role_permission` 本期**可暂不细化到按钮级权限**，先用"角色 = admin / reviewer"粗粒度门控即可（预留细粒度接口，见 §7）。

### 2.2 新增任务与报告表
```sql
CREATE TABLE review_task (
  id            BIGINT AUTO_INCREMENT PRIMARY KEY,
  task_no       VARCHAR(64)  NOT NULL,
  task_name     VARCHAR(255) NOT NULL,
  project_type_id BIGINT NULL,               -- 可空，接现有主数据可选
  secrecy_level_id BIGINT NULL,
  status        VARCHAR(32)  NOT NULL DEFAULT 'created',
                -- created→distributed→accepted→reviewing→done
  assignee_id   BIGINT NULL,                 -- 受理人(普通用户) FK sys_user
  distributed_by BIGINT NULL,                -- 分发操作人(管理员)
  distributed_at DATETIME(3) NULL,
  accepted_at    DATETIME(3) NULL,
  package_id     BIGINT NULL,                -- 预留：桥接现有 application_package
  created_by     BIGINT NULL,
  created_at     DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  updated_at     DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  KEY idx_assignee (assignee_id),
  KEY idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE task_report (
  id           BIGINT AUTO_INCREMENT PRIMARY KEY,
  task_id      BIGINT NOT NULL,
  report_type  VARCHAR(32) NOT NULL,         -- 5 类枚举，见 §1.1
  file_id      BIGINT NULL,                  -- FK file_object
  file_name    VARCHAR(255) NULL,
  review_status VARCHAR(32) NOT NULL DEFAULT 'pending', -- 供 5 审查按钮用
  uploaded_by  BIGINT NULL,
  uploaded_at  DATETIME(3) NULL,
  UNIQUE KEY uk_task_type (task_id, report_type),   -- 每任务每类型一份
  KEY idx_task (task_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
- 文件仍复用现有 `file_object` + `storage.py` 落盘逻辑（`backend/storage/`）。
- `task_report` 每任务固定 5 行（1 综合 + 4 专项），`review_status` 留给你后续的审查按钮更新。

### 2.3 `models.py` 同步
在 5A 段补 `SysUser.password_hash` 字段；在 5F 段新增 `ReviewTask`、`TaskReport` 两个 SQLModel 类，并加进 `__all__`。

## 3. 后端改动（FastAPI）

### 3.1 依赖新增
**无需新增第三方依赖**（决策：简单实现，只求区分用户功能）。用 Python 标准库即可：
- `hashlib`（sha256+盐）存密码，避免明文；
- `hmac`+`base64` 做签名 token（`user_id.role.exp` 载荷 + 签名），无需 JWT 库。

### 3.2 `app/security.py`（新，纯标准库）
- `hash_password / verify_password`（`hashlib.sha256(salt+pwd)`）
- `sign_token(user_id, role) / verify_token(token)`（`hmac` 签名 `payload`，密钥读 `settings.auth_secret`，在 `config.py` 加 `auth_secret` + `token_ttl_min`，默认给个内置值即可）。

### 3.3 `app/deps.py`（新）—— 认证/鉴权依赖
```python
get_current_user(Authorization: Bearer)      # 解析 token → 查 sys_user，401
require_role("admin") / require_role("reviewer")   # 403
```
- 兼容期：保留 `require_api_key` 用于 agent/隧道写库通道，两套不冲突。

### 3.4 路由（新）
| 文件 | 端点 | 权限 | 说明 |
|---|---|---|---|
| `routers/auth.py` | `POST /api/auth/login` | 公开 | 账号密码 → 返回 token+用户信息+角色 |
| | `GET /api/auth/me` | 登录 | 当前用户 |
| `routers/users.py` | `GET/POST /api/users`、`PATCH /api/users/{id}`、`POST /api/users/{id}/roles` | admin | **用户权限管理**：建号/停用/改角色/重置密码 |
| `routers/tasks.py` | `POST /api/tasks` | admin | **任务创建** |
| | `POST /api/tasks/{id}/reports`（multipart，可多次/批量） | admin | **报告上传 1+4** |
| | `POST /api/tasks/{id}/distribute {assignee_id}` | admin | **受理分发** |
| | `GET /api/tasks`（可筛选 status/assignee） | admin | **审查台账** |
| | `GET /api/tasks/{id}` | admin/受理人 | 任务详情（含 5 报告） |
| | `GET /api/my/tasks` | reviewer | **普通用户：我的任务** |
| | `POST /api/tasks/{id}/accept` | 受理人 | 普通用户受理确认（可选） |

- 在 `main.py` 的 `create_app()` 里 `include_router(auth/users/tasks, prefix="/api")`。
- 现有 6 个业务路由的写端点将来可逐步从 `require_api_key` 迁到 `get_current_user`（本期不强制，避免回归）。

### 3.5 种子脚本（新）`backend/scripts/seed_users.py`
- 写入 admin 账号（bcrypt hash）、reviewer 若干、`user_role` 绑定。避免明文 SQL。

## 6. 若选方案 A（复用现有表）的差异
- 任务=`application_package`，需先建 `declared_project`（放宽或预置 project_type/unit/person/secrecy 等必填 FK）。
- 分发：在 `application_package` 加 `assignee_id/assigned_at/assigned_by/accept_status` 四列。
- 报告=`material_file`，用 `material_category` 存 5 类；沿用现有上传/识别链路。
- 好处：直接接 `review_round/round_check`，审查按钮少写桥接；代价是任务创建表单重、改动面更大。

## 7. 工作量与顺序建议

| 阶段 | 内容 | 交付 |
|---|---|---|
| P1 认证地基 | 迁移 SQL(密码列/角色种子) + security/deps + auth 路由 + seed 脚本 + 前端登录/守卫 | 能登录、能识别角色 |
| P2 用户管理 | users 路由 + UserManagePage | 管理员建号/派角色 |
| P3 任务+报告 | review_task/task_report + tasks 路由(创建/上传) + TaskCreatePage | 管理员建任务传 1+4 |
| P4 分发+台账 | distribute + GET tasks + Distribute/Ledger 页 | 分发 + 台账 |
| P5 我的任务 | /api/my/tasks + MyTasksPage + TaskReviewPage(5 按钮占位) | 普通用户看到自己的任务 ✅ 目标达成 |

> 你要求的范围止于 P5「普通用户看到自己对应的任务」，5 个审查按钮内部逻辑不在本方案实现范围。

## 8. 决策（已定稿）
1. **任务建模**：✅ 方案 B 轻量新表（`review_task` + `task_report`，预留 `package_id`）。
2. **登录鉴权**：✅ 简单实现——固定用户名+密码，标准库 `hashlib` 存密码 + `hmac` 签名 token；重点是按角色区分功能，不引入 JWT/bcrypt 重依赖。
3. **权限粒度**：✅ 角色粗粒度（admin / reviewer）；`permission` 表预留待后续细化。
4. **受理环节**：✅ 分发即受理——`distribute` 设 `assignee_id` 即进入该用户"我的任务"，`status='distributed'`，无独立 accept 步骤（去掉 `accepted_at` 也可，保留亦无妨）。
