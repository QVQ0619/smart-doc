## 5A 用户权限域（谁能用、能用什么）

| 表名 | 中文 | 存储内容 / 名字含义 |
|---|---|---|
| `org` | 机构 | 受理机构、依托单位等单位树。org = organization，自引用形成上下级 |
| `sys_user` | 系统用户 | 登录账号、所属机构、可访问的最高密级。sys_user = system user |
| `role` | 角色 | 角色定义（系统管理员/审查员等）。role = 角色 |
| `user_role` | 用户-角色 | 用户与角色的多对多绑定。中间关联表 |
| `permission` | 权限点 | 细粒度权限项及其作用域（全局/基金/批次）。permission = 权限 |
| `role_permission` | 角色-权限 | 角色与权限点的多对多绑定。中间关联表 |

## 5B 依据库主数据域（规则从哪来、长什么样）

| 表名 | 中文 | 存储内容 / 名字含义 |
|---|---|---|
| `review_dimension` | 审查维度 | 六维（完整性/规范性/合规性/一致性/合理性/真实性）。dimension = 维度 |
| `standard_doc` | 规则标准文件 | 政策/指南文件的元数据（标题、发文机构、年份），原文件进对象存储。standard_doc = 标准文档 |
| `regulation_clause` | 依据条款 | 从标准文件拆出的一条条法规条款原文。regulation clause = 法规条款 |

| `review_rule` | 审查规则（身份） | 规则的“身份证”：规则编号 + 指向当前生效版本。只是壳，内容在版本表 |

| `rule_stage` | 规则-阶段关联 | 规则只适用于某些阶段。中间表 |
| `rule_param` | 规则参数 | 参数化规则在不同项目下的阈值（如“年龄≤X”）。param = parameter |
| `form_template` | 申报书字段模板 | 某项目×阶段的申报书有哪些字段。form = 表单 |
| `form_field` | 字段定义 | 模板下每个字段的定义（字段码、类型、是否必填）。field = 字段 |
| `report_template` | 报告模板 | 审查报告的排版模板文件。report template = 报告模板 |
| `required_material_item` | 必交材料项 | 某项目×阶段规定必须交哪些材料/附件。required material = 必交材料 |

## 5C 配置包域（“换包不换引擎”的核心）

## 主线串联（一句话看懂全图）

> `declared_project`（报了什么）→ 装进 `review_batch`（哪一批，挂 `config_package` 规则包）→ 形成 `application_package`（材料包）→ `material_file` / `parse_segment`（拆解材料）→ 结构化成 `extracted_field` / `budget_item` 等 → `review_round` + `round_check`（逐条审查，真实性维度触发 `sim_run` 仿真验真）→ `review_report`（出报告）→ `review_case`（沉淀案例）。
>
> 文件都登记在 `file_object`，全程操作记到三张日志表（`agent_run_log` / `model_call_log` / `audit_log`）。

**两个全局锚点**：
- 溯源锚 `parse_segment`：任何审查发现都能溯回原文页定位。
- 版本锚 `review_rule_version`：规则不可变版本，历史审查永远可复现。
