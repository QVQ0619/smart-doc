# 形式审查 · 子项目 A：申请材料上传 + 识别 — 设计文档

> 日期：2026-06-25　状态：设计已确认（待用户复审 spec）
> 所属：「立项审查 AI 辅助系统」之**形式审查**能力（系统三环节的第二环：规则准备✅ → **形式审查⏳** → 专家审查⏳）

## 1. 背景与定位

`docs/系统提示词.md` 已把**形式审查**定义为：依据**「硬性(hard)」规则**，逐项核对申请材料的齐全性、规范性、资格符合性；客观判定；每条输出**四要素**（结论 / 依据 / 出处 / 建议），最后按维度汇总。

后端 schema（`审查系统-schema_mysql.sql`，56 表）已为这条流水线设计好全部表，目前**全空、零业务代码、零 seed**。本期把整条形式审查流水线按**水平阶段**拆为 4 个子项目，逐个跑通：

- **A 申请材料上传 + 识别**（本文）
- B 结构化抽取（申请人/经费/附件…入对应表）
- C 形式审查执行（按配置包逐条 hard 规则判定 → `round_check` + `finding_evidence`）
- D 审查报告（汇总 `review_report`）

**驱动架构**：Blade agent + skill（延续规则准备侧：cloudflared 隧道 + X-API-Key 写鉴权 + python shim + BackgroundTasks 异步识别）。agent 负责后续 C 的四要素判定；后端提供幂等能力端点，skill shim 调用。

## 2. 本期目标与验收

**目标**：用户在 Blade 聊天上传一份 PDF/DOCX（或扫描图片）申请书并说"这是待审查的申请材料" → agent 经 skill `save-material-doc` 把文件推到本地后端 → 后端入 `material_file` + 自动造占位父行 + 识别成 `parse_segment`（扫描件 OCR 兜底）；前端「审查材料」只读页能看到该审查包的材料、识别状态与段落。

**验收标准**：
1. 后端 `POST /api/material-files` 上传一份 .docx/.pdf → 返回 `material_file_id` + `package_id`，识别后 `parse_segment(material_file_id=…)` 落库、段落数 > 0。
2. 同一 `package_id` 再传第二个文件 → 挂到同一审查包（一个包多文件）。
3. 前端「审查材料」页：列审查包 → 展开看其 material_files + 识别状态 → 再展开看 parse_segment。
4. 后端全量测试绿；前端 vitest 绿 + `npm run build` 零错误。

**非目标（YAGNI，留给 B/C/D 或更后）**：结构化抽取、审查判定、报告；申报批次管理 UI；多轮次（round_no 恒为 1）；附件类型/密级细分；material_category 智能分类（本期由调用方给或默认）。

## 3. 数据模型与占位父行策略

`material_file` 必须挂在 `application_package` 上，而 `application_package` 强制要求 `batch_id`(review_batch) 与 `declared_project_id`(declared_project)，两者又各有一串 NOT NULL 外键。由于 schema 零 seed，占位父行的**全部前置主数据**也必须由后端**幂等创建**。

**FK 依赖链（最小桩需创建的行）**：

| 表 | 关键 NOT NULL 字段 | 占位取值（哨兵 `code="__DEFAULT__"` get-or-create） |
|---|---|---|
| `project_type` | code/name/sector | `__DEFAULT__` / "默认项目类型" / "默认" |
| `review_stage` | code/name | `__DEFAULT__` / "形式审查" |
| `secrecy_level` | code/name/rank | `__DEFAULT__` / "公开" / 0 |
| `research_unit` | name | "默认依托单位"（按 name 去重） |
| `research_person` | name, unit_id | "默认申请人", ↑unit |
| `sys_user` | username | `__system__`, display_name="系统"（org_id 留空） |
| `review_batch` | batch_no, project_type_id, stage_id, created_by | `batch_no="__DEFAULT_BATCH__"` 复用同一行 |
| `declared_project` | project_code, project_name, project_type_id, declaring_unit_id, applicant_person_id, secrecy_level_id | 每个审查包新建一条（project_code 用包序列） |
| `application_package` | batch_id, declared_project_id | 新建（本期审查包主体） |

**策略**：实现一个 `ensure_default_master_data(db) -> DefaultRefs` 幂等帮助函数——按哨兵 `code` / 固定 name 查，缺则建，返回 6 类主数据 + 默认 batch 的 id。每次新建审查包时复用主数据与默认 batch，仅新建 `declared_project` + `application_package`。

**审查包语义**：一个 `application_package` = 一份申请，可含多个 `material_file`。`POST /api/material-files` 的可选入参 `package_id` 决定是新建包还是挂到既有包。

## 4. 后端接口

### 4.1 `POST /api/material-files`（写，`Depends(require_api_key)`）

借鉴 `routers/standard_docs.py:42` 的 `upload_standard_docs` 写法（UploadFile + BackgroundTasks + FileStorage）。

- 入参：`files: list[UploadFile]`；表单 `package_id: int | None`（可选）；`material_category: str | None`（可选，默认 "申请书"）。
- 逻辑：
  1. `ensure_default_master_data(db)`；
  2. `package_id` 缺 → 新建 `declared_project` + `application_package`（status="parsing"）；传了 → 校验存在，复用；
  3. 对每个上传文件：存盘 `FileStorage` → `file_object` → `material_file`(package_id, round_no=1, material_category, file_name, file_format=后缀, file_id, secrecy_level_id=默认, recognition_status="pending")；
  4. 每个 material_file 触发后台识别（BackgroundTasks，先置 "processing"）。
- 返回 `MaterialUploadResult`：`package_id` + `items:[{material_file_id, file_name, recognition_status}]`。
- 复用 `FileTooLargeError` 等既有错误处理。

### 4.2 `GET /api/material-packages`（读，无鉴权，与现有 GET 一致）

返回审查包列表，每个含其 material_files 概要：`[{package_id, created_at, file_count, files:[{material_file_id, file_name, material_category, recognition_status, segment_count}]}]`。`segment_count` 由 `parse_segment` count 派生。

### 4.3 `GET /api/material-files/{id}/segments`（读，无鉴权）

返回该 material_file 的 `parse_segment` 列表（page_no/locator/segment_type/content_text），供前端展开与 shim 轮询。

### 4.4 `POST /api/material-files/{id}/recognize`（写，`require_api_key`）

手动重识别单个材料（镜像 `standard_docs/{id}/recognize`），便于失败重试。

## 5. 识别核心重构（共享解析）

`recognition.py` 中 `parse_pdf/parse_docx/parse_image` 已通用（返回 `SegmentDraft`），仅 `recognize_standard_doc` 是文档专属。重构：

- 抽出 `parse_file(path, ext) -> (drafts, error)`：集中"按扩展名选 parser + 空结果错误文案"逻辑（即现 `recognize_standard_doc` 第 130–162 行的 try 块）。
- `recognize_standard_doc` 改为调用 `parse_file` + 写 `parse_segment(standard_doc_id=…)`（行为不变，回归测试守住）。
- 新增 `recognize_material_file(db, storage, material_file_id) -> RecognizeResult`：调用 `parse_file` + 写 `parse_segment(material_file_id=…, standard_doc_id=None)`；segment_type 同样经 `_SEGMENT_TYPE_MAP` 映射到 DB 白名单（title/text/table/figure）；置 `material_file.recognition_status`/`recognition_error`。
- `reset_stuck_processing` 扩展：同时把 `material_file` 残留 "processing" 重置为 "pending"（应用启动时，main 已调用一次）。

## 6. Skill `save-material-doc`

`blade/skills/save-material-doc/`，结构对齐 `save-rule-doc`（记忆 [[skill-dir-blade-coa-layout]]：`versions/1.0.0/SKILL.md` + `scripts/*.py` shim + `metadata.json`；推送 payload 扁平）。

- 触发：agent 在用户上传文件并表达"这是待审查的申请材料"时调用。
- shim：经 cloudflared 隧道 POST 文件到本地 `POST /api/material-files`（带 X-API-Key），可带上一文件返回的 `package_id` 以归入同一份申请；轮询 `GET /api/material-files/{id}/segments` 回报识别结果（段落数）。
- 忠实回报：入库 material_file、识别段落数、失败原因如实转述（顶层铁律 5）。

## 7. 前端「审查材料」只读页

## 8. 错误处理

- 不支持的格式 / 扫描件无 OCR / 空文档：复用 `recognition.py` 既有错误文案，落 `material_file.recognition_error`，状态 "failed"。
- `package_id` 传了但不存在 → 404。
- 上传无 X-API-Key → 401（鉴权依赖）。
- 前端：列表 isError / 段落 isError / 空状态分别处理。
