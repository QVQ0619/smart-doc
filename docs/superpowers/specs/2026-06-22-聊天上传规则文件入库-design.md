# 聊天上传规则文件入库（Blade agent → 规则库）Design

> 立项审查 AI 辅助系统的子功能。前序：前端骨架（2026-06-18）、文件存储后端 v2（2026-06-18，已完成）。

## Overview

让用户在右侧 Blade 聊天窗口附带上传一个规则文件并发指令（如「把这个存为规则文件」），由 Blade agent 识别这是规则文档后，**把原文件直接存入规则库**（写 `file_object`+`standard_doc`，二进制落本地磁盘），无需用户离开聊天去规则库页手动上传。

## Goal

打通「聊天附件 → agent 识别 → 原文件入规则库」的链路，复用已完成的文件存储后端，**后端零改动**。本期只做「识别后原文件直接入库」，不做内容提取/结构化。

## 背景与既有能力

- **聊天**：右栏用 Blade SDK 预制组件 `<ChatView>`（`@blade-hq/agent-kit@0.5.11`），**原生支持输入框附带文件**（`FileComposerAttachment`）。会话连 Blade 平台，附件经 Blade 原生接口 `POST /api/sessions/{id}/upload/{dir}` 落到 agent 的 session sandbox。
- **后端**：本地 FastAPI（`:8000`）已实现 `POST /api/standard-docs`（multipart，字段名 `files`）→ 持久化为 `file_object`(bucket="local") + `standard_doc`，二进制存 `backend/storage/standard_doc/`。返回：
  ```json
  {"uploaded":[{"id","doc_code","title","file_name","size_bytes","mime_type","created_at"}], "failed":[{"name","reason"}]}
  ```
  整体 200；单文件超限/失败归入 `failed[]`。
- **参考工程** `E:\shixi\blade-coa-master`：Blade solution 工程，模式为「后端把 CLI 暴露成 HTTP（`router_cli.py` 的 `POST /cli`）+ agent 容器挂薄 shim（`tools/fires-cli-shim/fires-cli`）转发 + skill(markdown) 指挥 agent 何时调用」。

## 关键决策（brainstorm 确认）

1. **触发方式**：用户发指令让 agent 保存（非自动、非纯上传入口）。
2. **Blade 配置权**：用户可在 Blade solution 侧加 skill / 配置 agent。
3. **网络拓扑**：agent sandbox 与 smart-doc 后端**同机/同网，agent 能 HTTP 访问后端**（但不保证共享文件系统 → 文件字节须经 HTTP 传输）。
4. **「处理」含义**：识别后**原文件直接入库**（复用现有接口，后端不改）。
5. **方案**：方案 A —— 薄上传 shim + 复用现有接口（否决了 B「完整 CLI/`/cli`」因 argv 协议搬不了文件字节、属过度设计；否决了 C「前端居中拦截」因不符合 agent/CLI 设想且拓扑使其唯一优势失效）。
6. **规则库页刷新**：轮询，组件挂载时每 10 秒 refetch。

## 架构与组件

| 组件 | 位置 | 职责 | 状态 |

## 数据流

```
用户在聊天框附带 政策A.pdf + "把这个存为规则文件"
  ▼ Blade 原生把附件上传到 session sandbox
agent 加载 save_rule_doc skill → 确认是规则文件 → 跑 `smart-doc-add <sandbox路径>`
  ▼ shim 读文件 → multipart POST http://<后端>/api/standard-docs
后端持久化（file_object + standard_doc）→ 返回 uploaded[]
  ▼ shim 打印 doc_code=SD-xxxx title=政策A（exit 0）
agent 回报："已存入规则库：政策A（SD-xxxx）"
  ▼（每 10s 轮询）规则库页出现新行
```

## 组件设计

### 1. `smart-doc-add` shim（纯 stdlib，仿 fires-cli shim）

- **源码**：`backend/agent_shim/smart_doc_add.py`，仅用标准库（`urllib`、手搓 multipart/form-data），无第三方依赖。结构化为可测函数：`build_multipart(files) -> (body, content_type)`、`upload(api_base, paths, timeout) -> result_dict`、`main(argv) -> int`。
- **用法**：`smart-doc-add <path> [<path> ...]`（可多文件，对应后端多文件上传）。
- **环境变量**：
  - `SMART_DOC_API`（必填）：后端 base URL，例 `http://smart-doc-backend:8000`。未设置 → stderr 报错 + 退出码 6。
  - `SMART_DOC_TIMEOUT`（可选，默认 120）：单次请求最大等待秒数。
- **输出**：stdout 对每个 `uploaded` 项打印 `doc_code=<code> title=<title> file=<file_name>`；stderr 对每个 `failed` 项打印 `FAILED <name>: <reason>`。
- **退出码约定**（让 agent 准确判断）：
  | 码 | 含义 |
  |---|---|
  | 0 | 全部入库成功 |
  | 2 | 入参文件不存在/不可读 |
  | 3 | 网络错误（连不上后端） |
  | 4 | 后端返回非 2xx |
  | 5 | 后端 2xx 但返回里有 `failed` 项（部分或全部失败） |
  | 6 | `SMART_DOC_API` 未配置 |

### 2. `save_rule_doc` skill（markdown）

参照 blade-coa skill 结构，含：
- **何时用**：用户在聊天里附带了文件，且表达「存为规则文件 / 加入规则库 / 保存到规则库」等意图。
- **步骤**：① 判断是否规则类文档（政策/指南/规范）；② 在 sandbox 定位刚上传的文件路径；③ 跑 `smart-doc-add <路径>`；④ 读退出码 + stdout；⑤ 据实回报——成功报 `标题(doc_code)`，失败报对应原因。
- **护栏**：意图不明确先问、不擅自入库；判断非规则文档时说明并询问是否仍保存；**仅当退出码 0 且有 uploaded 才可声称成功**，失败不许谎报。

### 3. 规则库页轮询（前端小改）

`src/components/StandardDocLibrary.tsx` 中的 `useQuery({ queryKey:["standard-docs"], queryFn: listStandardDocs })` 增加 `refetchInterval: 10000`。组件仅在「规则库」页挂载时存在，故只在该页打开时每 10s 轮询，切走自动停。不引入轮询时的 loading 抖动（react-query 后台 refetch 默认静默）。

### 4. 后端

不改动。现有 `POST /api/standard-docs` 即 agent 入库的唯一依赖。

## 错误处理

- **shim**：按 §1 退出码 + stderr 明确信息分类（文件缺失/网络/后端错/部分失败/缺配置）。
- **后端**：超限/单文件失败已归入 `failed[]`（整体 200），不改。
- **skill**：明确要求 agent 按退出码如实回报，失败不谎报成功。

## 测试策略

- **shim 单测** `backend/tests/test_agent_shim.py`（用后端 venv + pytest）：
  - `build_multipart` 构造正确（边界、字段名 `files`、文件名/字节）。
  - `main` 成功路径：起本地 stub HTTP server（`http.server` 后台线程，返回固定 `uploaded`）→ exit 0 + stdout 含 doc_code。
  - 文件不存在 → exit 2；后端 500 → exit 4；返回含 `failed` → exit 5；`SMART_DOC_API` 缺失 → exit 6。
  - 用 `monkeypatch.setenv` 控制 `SMART_DOC_API` 指向 stub。
- **后端**：零改动，现有 15 测试继续覆盖持久化。
- **前端**：`StandardDocLibrary.test.tsx` 现有 3 测试不回归；轮询为静默后台行为，按需补一个「query 配置含 refetchInterval」的轻量断言（避免 fake timers 复杂度）。
- **skill / 端到端**：markdown 无法单测，靠人工 e2e（见下）。

## 范围边界（务必明确）

- **本仓库产出**：① `smart-doc-add` shim（`backend/agent_shim/smart_doc_add.py`）+ 单测；② `blade/skills/save_rule_doc/SKILL.md`；③ `StandardDocLibrary` 轮询小改 + 测试；④ 部署说明 `blade/README.md`。
- **用户在 Blade 侧做**（不在本仓库）：把 shim 挂进 agent 容器并暴露为 PATH 上的 `smart-doc-add`、设 `SMART_DOC_API`/`SMART_DOC_TIMEOUT`、把 `save_rule_doc` skill 装进 solution、确认聊天附件上传已开启。

## 部署说明（`blade/README.md` 要点）

1. 把 `backend/agent_shim/smart_doc_add.py` 挂入 agent 容器（建议暴露为可执行 `smart-doc-add`：加 shebang + chmod +x，或包一层 `#!/usr/bin/env python3` 包装脚本）。
2. 在 agent 容器设 `SMART_DOC_API=http://<后端可达地址>:8000`（同机/同网地址，非 localhost——除非 agent 与后端真正同主机网络命名空间）。
3. 把 `blade/skills/save_rule_doc/` 装进你的 Blade solution 的 skill 集。
4. 确认 `<ChatView>` 附件上传开启（SDK 原生）。

## 端到端人工验证（需真实 agent + 真实 smart 库）

1. 起后端（`backend/`）：`.venv/Scripts/python -m uvicorn app.main:app --reload --port 8000`。
2. 起前端：`npm run dev` → 进任意页（右栏聊天可见），最好停在「规则库」页。
3. 聊天框附带 `政策A.pdf` + 发「把这个存为规则文件」。
4. agent 回报「已存入规则库：政策A（SD-xxxx）」。
5. 规则库页 10s 内自动出现新行；查 `smart.standard_doc`/`smart.file_object` 有行、磁盘 `backend/storage/standard_doc/` 有文件。
6. 反例：附带一个明显非规则文档 + 含糊指令 → agent 应先询问/说明，不擅自入库。

## 验收标准

1. ✅ 聊天附带文件 + 指令 → agent 调 `smart-doc-add` → 原文件入 `file_object`+`standard_doc`、磁盘有文件。
2. ✅ shim 退出码语义正确（成功/文件缺失/网络/后端错/部分失败/缺配置），agent 据实回报、失败不谎报。
3. ✅ 规则库页打开时每 10s 轮询，agent 入库后自动出现新行。
4. ✅ 后端零改动；现有后端 15 + 前端 26 测试不回归；新增 shim 单测全绿。
5. ✅ 仓库内交付 shim + skill + 部署说明 + 前端轮询；Blade 侧部署步骤文档化。

## Out of Scope / YAGNI

- 内容提取/结构化/摘要（本期只存原文件）。
- 完整 CLI 框架 + `/cli` HTTP 路由（方案 B，过度设计）。
- 前端居中拦截 tool-call（方案 C）。

- 聊天里删除/管理规则文件（仍走规则库页）。
