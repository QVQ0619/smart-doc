# 规则抽取（子项目2）设计

> 立项审查 AI 系统「文档识别 + 规则抽取」功能的**第二个子项目**。
> 消费子项目1 产出的 `parse_segment`，产出 `regulation_clause`（带出处）。

## 目标

用户在聊天里让 Blade agent 抽取某**已识别**规则文档的审查条款；agent 经后端读该文档的
`parse_segment`、判定哪些是规则条款、带出处写入 `regulation_clause`；规则库页可展开行
按文档归类展示抽取出的条款（条号、条文、第几页第几段出处）。

## 背景与既有事实

- **子项目1 已交付**（HEAD c114677）：规则文件入库后自动识别成 `parse_segment`
  （`standard_doc_id` 设值、`material_file_id=NULL`、`page_no`/`locator`/`segment_type`/
  `content_text`）。⚠️ **`parse_segment.segment_type` 是 VARCHAR(8) CHECK
  `IN ('text','table','title','figure')`**；识别阶段已把解析器的 heading→title、
  paragraph→text、table→table 映射入库。**agent 读到的 segment_type 是 title/text/table**。
- **`regulation_clause` 表**（schema 5.9，已建）：
  `id`、`standard_doc_id`(FK→standard_doc，可空)、`doc_code`(VARCHAR40 **NOT NULL**)、
  `clause_no`(VARCHAR40 **NOT NULL**)、`clause_text`(TEXT 可空)、`source_path`(VARCHAR500 可空)、
  `source_segment_id`(FK→parse_segment，可空)，外加 embedding 列(本子项目留空)。
  **无 UNIQUE(doc, clause_no)** ——幂等替换靠 delete-by-standard_doc_id + insert。
- **出处链**：`regulation_clause.source_segment_id` → `parse_segment.page_no` + `locator`。
- **会话推送 skill 模式**（子项目「会话推送入库」已落地）：`src/blade/sessionSkill.ts` 用
  `partnerSkillApi.uploadSessionSkill(sessionId, {name, files})` 把本地 skill 推到远端 agent
  会话沙箱，agent 自动发现。技能名须匹配 `^[a-z0-9-]+/[a-z0-9-]+$`。现已推
  `local/save-rule-doc`（`blade/skills/save-rule-doc/`：SKILL.md + scripts/ + api_base.txt）。
- **可达性**：远端 agent 经 cloudflared 隧道访问本地后端；`api_base.txt` 由前端
  `VITE_SMART_DOC_API` 注入。
- **测试基线**：后端连真实 MySQL `smart_test`（conftest 用 schema SQL 建库；清理需先删子表）。

## 范围

### 本子项目做
1. 后端 3 接口：GET 段落（供 agent 读）、POST 条款（幂等写 regulation_clause）、GET 条款（带出处，供前端）。
2. skill② `extract-review-rules`（会话推送）+ 两个 shim。
3. `sessionSkill.ts` 扩展为推送两个 skill。
4. 前端规则库可展开行展示条款。

### 本子项目不做（明确推迟）
- **`review_rule` / `review_rule_version` 结构化**（维度/decision_type/disposition/binding_class
  分类、`review_rule_clause` 关联）——**子项目3**。本子项目只产出 `regulation_clause`。
- **embedding/向量化**（regulation_clause 的 embedding 列留 NULL）。
- **schema 变更**：不新增任何列（DB 预建、权威）。「是否已抽取」由该文档 regulation_clause
  行是否存在判断，不加 extraction_status。
- **抽取质量保证/完备性**：抽取是 agent 判断，不保证抽全；靠铁律确保只如实回报实际入库条数。

## 架构与数据流

```
聊天「抽取《申请规定》的审查规则」
  └─ skill② extract-review-rules（会话推送，同 save-rule-doc 机制）
       1. 解析目标 doc_id：smart_doc_segments 接受 doc_id/doc_code/标题，内部经 GET 列表解析
          （纯数字→id；SD-开头→doc_code；否则→标题子串。标题命中多条→列候选并非零退出，agent 据此向用户澄清）
       2. GET /api/standard-docs/{id}/segments → 该文档全部 parse_segment(id/page_no/locator/segment_type/content_text)
       3. ★agent 判定★：哪些段是规则条款 → 产出 [{clause_no, clause_text, source_segment_id}]
       4. agent 把结果写成 JSON 文件 → shim POST /api/standard-docs/{id}/clauses
       5. 后端幂等替换写 regulation_clause（补 standard_doc_id/doc_code、校验 source_segment_id）
       6. agent 回报「已抽取 N 条规则」（铁律：只转述 shim 实返条数）
  └─ 前端规则库行展开 → GET /api/standard-docs/{id}/clauses → 列出条号/条文/出处
```

## 组件

### 后端服务 `backend/app/extraction.py`（新建）
```python
def replace_clauses(db: Session, doc_id: int, clauses: list[ClauseIn]) -> ClauseWriteResult:
    """幂等：删该 standard_doc 旧 regulation_clause → 插新。
    后端补 standard_doc_id=doc_id、doc_code=standard_doc.doc_code。
    校验每条 source_segment_id 必须是本文档的 parse_segment；非法→该条 source_segment_id 置 NULL，
    并计入 missing_provenance（不整批失败）。clause_text 为空也允许（clause_no NOT NULL）。
    单次 commit。返回 ClauseWriteResult(inserted, missing_provenance)。"""
```
- 调用方先保证 doc 存在（路由层 404）；本函数假定 doc 存在。
- 合法 source_segment_id 集合 = `SELECT id FROM parse_segment WHERE standard_doc_id=doc_id`。

### 后端路由 `backend/app/routers/clauses.py`（新建，注册到 /api）
- **`GET /api/standard-docs/{doc_id}/segments`** → `list[SegmentOut]`；doc 不存在/软删→404。
- **`POST /api/standard-docs/{doc_id}/clauses`** → body `ClauseBatchIn` → 调 `replace_clauses` →
  `ClauseWriteResult`；doc 不存在/软删→404；空 clauses 列表 → inserted=0（不报错）。
- **`GET /api/standard-docs/{doc_id}/clauses`** → `list[ClauseOut]`，**LEFT JOIN parse_segment**
  取 `page_no`/`locator`（按 clause_no 或 id 排序）；doc 不存在/软删→404。

### 后端 schemas `backend/app/schemas.py`（追加）
```python
class SegmentOut(BaseModel):
    id: int
    page_no: Optional[int]
    locator: Optional[Any]
    segment_type: str
    content_text: Optional[str]

class ClauseIn(BaseModel):
    clause_no: str
    clause_text: Optional[str] = None
    source_segment_id: Optional[int] = None

class ClauseBatchIn(BaseModel):
    clauses: list[ClauseIn]

class ClauseWriteResult(BaseModel):
    inserted: int
    missing_provenance: int      # source_segment_id 非法被置 NULL 的条数

class ClauseOut(BaseModel):
    id: int
    clause_no: str
    clause_text: Optional[str]
    source_segment_id: Optional[int]
    page_no: Optional[int]       # 来自 join parse_segment
    locator: Optional[Any]
```

### skill② `blade/skills/extract-review-rules/`（会话推送，新建）
**结构沿用 `save-rule-doc`（SKILL.md + scripts/，连字符名 `extract-review-rules`，无 `-skill`
后缀、不生成 AGENTS.md/install.sh/evals——那是从零造独立可装 skill 才用；本 skill 走 Blade
会话推送，须与 save-rule-doc 同构才能被 `uploadSessionSkill` 推送/被 agent 发现）。但内容
遵循 agent-skill-creator 的质量规范：清晰的触发 description、无占位符/TODO、脚本功能完整。**

- `SKILL.md`：触发（用户说「抽取……审查规则」并指明/暗示目标文档）；步骤（解析 doc_id→GET 段落
  →判定条款→写 JSON→POST→回报）；铁律（只转述 shim 实返 inserted 条数，绝不编造条数/条文；
  缺出处如实报 missing_provenance；不绕过 shim 直连 DB；目标文档未识别成功时先提示而非硬抽）。
- `scripts/smart_doc_segments.py`（纯标准库）：`<doc_id|doc_code|标题>` → 先 GET
  `/api/standard-docs` 解析出唯一 doc_id（纯数字直用；`SD-`开头按 doc_code 精确匹配；否则按标题
  子串匹配。**标题命中 0 条→退出码 8 并提示；命中多条→打印候选(id/doc_code/标题)并退出码 7**，
  agent 据此请用户用 doc_code 指明）→ 再 GET `/api/standard-docs/{id}/segments` → stdout
  **首行打印 `doc_id=<n>`**，随后打印段落 JSON 供 agent 读。退出码：0 成功 / 3 网络 / 4 非2xx /
  6 缺 SMART_DOC_API / 7 标题歧义 / 8 未找到。
- `scripts/smart_doc_clauses.py`（纯标准库）：`<doc_id> <clauses.json 路径>`（doc_id 用上一步
  首行解析出的数字）→ 读 JSON（形如 `{"clauses":[{"clause_no","clause_text","source_segment_id"}]}`）
  → POST `/api/standard-docs/{id}/clauses` → 打印 `inserted=<n> missing_provenance=<m>`。
  退出码：0 / 2 文件不存在 / 3 网络 / 4 非2xx / 6 缺 SMART_DOC_API。
- `scripts/api_base.txt`：后端地址（前端注入，同 save-rule-doc）。

### 前端会话推送 `src/blade/sessionSkill.ts`（修改）
扩展为**推送两个 skill**：除现有 `local/save-rule-doc` 外，新增推送
`local/extract-review-rules`（SKILL.md + 两个 shim + api_base.txt）。两次推送都 best-effort
try/catch 不抛、不 await（沿用现状）。`?raw` 引入新 SKILL.md 与两个 shim 文本。

### 前端展示 `src/components/StandardDocLibrary.tsx` + `src/api/standardDocs.ts`（修改）
- API 客户端加 `listClauses(docId): Promise<Clause[]>`（GET `/api/standard-docs/{id}/clauses`）；
  `interface Clause{id, clause_no, clause_text, source_segment_id, page_no, locator}`。
- Table 加 `expandable.expandedRowRender`：展开某文档行 → useQuery 拉 `listClauses(row.id)` →
  子表列出 `clause_no`、`clause_text`、出处（`page_no!=null` 显示「第X页」+ locator 的
  block_index/para_index 显示「第Y段」；docx 无页码只显示段）。空 → 提示「尚未抽取，可在聊天里
  让 AI 抽取本文档规则」。

## 出处与校验
- 出处渲染：`page_no` 非空 → 「第{page_no}页第{block_index|para_index}段」；`page_no` 空（docx）
  → 「第{para_index}段」；无 locator 索引 → 仅页或「—」。
- POST 校验：`source_segment_id` 不在本文档 parse_segment 集合内 → 置 NULL，计入
  `missing_provenance`，不使整批失败。`clause_no` 由 agent 提供（NOT NULL）；约定 agent 从文本取
  「第三条」「3.1」等，取不到用顺序兜底（如 `#1`、`#2`）。`doc_code` 后端从 standard_doc 填。

## 错误处理
- doc 不存在/软删（is_active=False）→ 三个接口均 404。
- POST 空 clauses → inserted=0、missing_provenance=0（不报错）。
- 目标文档 `recognition_status != done`（无 parse_segment）→ GET segments 返回空列表；SKILL.md
  指示 agent 先提示「该文档尚未识别成功，无法抽取」，不硬抽、不编造。
- 目标文档解析：标题命中多条（如多个同名「申请规定」）→ smart_doc_segments 退出码 7 + 列候选，
  SKILL.md 指示 agent 请用户用 doc_code 指明，不擅自猜；命中 0 条→退出码 8，提示未找到。
- shim 网络/非2xx → 退出码区分，SKILL.md 据码如实回报，绝不谎报成功。

## 测试策略（后端连 MySQL smart_test + TestClient；前端 vitest）
- **`replace_clauses` 服务测**：构造 doc + 若干 parse_segment → 传 clauses（含一个非法
  source_segment_id）→ 断言 regulation_clause 行数=inserted、标准doc_id/doc_code 正确、非法
  source_segment_id 被置 NULL 且 missing_provenance 计数正确；**幂等**：再调一次断言替换不累加。
- **接口测**：GET segments 返回该文档段落（不含别的文档）；POST clauses 写入并返回计数；
  GET clauses join 出处（page_no/locator 来自 parse_segment）；doc 不存在→404；软删→404；空 clauses→inserted=0。
- **shim 测**（stub HTTPServer）：`smart_doc_segments` GET 打印 JSON、退出码；`smart_doc_clauses`
  读 JSON 文件 POST、打印 inserted/missing_provenance、退出码（缺文件→2、缺 env→6、非2xx→4）。
- **前端测**：`listClauses` 发对 GET；展开行渲染条款与出处（含 docx 无页码只显示段的分支）；
  空条款显示提示。全量 `npm test` 通过、`npm run build` 零类型错误。

## 验收标准
1. 聊天里对一个已识别文档说「抽取审查规则」，agent 经 skill② GET 段落、判定后 POST，
   `regulation_clause` 出现该文档多条（`standard_doc_id`/`doc_code` 正确、`source_segment_id`
   指向本文档 parse_segment）。
2. 重新抽取同一文档 → 条款被替换而非累加（幂等）。
3. 含非法 source_segment_id 的提交 → 该条入库但 source_segment_id=NULL，响应 missing_provenance≥1，
   agent 如实回报。
4. 规则库页展开某文档行 → 列出该文档条款（条号、条文、第X页第Y段出处）；未抽取的文档展开显示提示。
5. agent 回报的条数与后端实际入库条数一致（铁律，无夸大/编造）。
6. 目标文档未识别成功（无 parse_segment）时，agent 提示无法抽取而非硬抽/编造。
7. 后端全量 pytest 通过；前端 npm test 全绿、npm run build 零类型错误。
