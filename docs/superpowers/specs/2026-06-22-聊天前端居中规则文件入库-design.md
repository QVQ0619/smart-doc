# 聊天前端居中规则文件入库（P-B）Design

> 立项审查 AI 辅助系统子功能。承接「聊天上传规则文件入库」需求，但因部署现实改走**前端居中**方案。前序：前端骨架、文件存储后端 v2（均完成）、save_rule_doc（shim+skill，已完成但因下述原因暂不部署）。

## Overview

用户在右栏 Blade 聊天面板的「规则文件入库」操作条选一份规则文件 → 前端把它传到 agent 的会话 sandbox 并自动发消息让 **agent 在对话里识别/建议** → 用户看完建议点「存入规则库」→ **前端**用浏览器直接把文件存入本地规则库（`POST /api/standard-docs` → `file_object`+`standard_doc`）。

## Goal

在「远程托管 agent + 本地后端、且不能自托 agent 镜像」的现实约束下，实现「聊天里 agent 协助识别、用户确认、文件入本地规则库」的体验。**后端零改动、Blade 侧零配置**。

## 为什么是 P-B（约束沿革）

原设想（save_rule_doc）让 agent 经 shim 调后端入库，前提是 agent 与后端**同机/同网可 HTTP 互通**——这要么自托一套 blade-coa 式 compose（含 blade-agent 容器），要么把后端公网暴露。经核实：

- **拿不到自托 agent 镜像**：`registry.cn-beijing.aliyuncs.com/bladeai/blade-agent` 等 ACR 镜像登录被拒（`unauthorized`），无法自托 → blade-coa 式 compose 部署不可行。
- agent 实际在**远程**（`http://115.190.152.1:8020`），其 sandbox **够不到**本地 `localhost:8000` 后端与本地 `smart` 库。

因此走 **P-B 前端居中**：agent 保持远程，**浏览器**作桥（浏览器同时够得着远程 Blade 和本地后端）。真正执行入库的是前端；agent 只做对话内的识别/建议。否决 P-A（公网暴露后端，安全顾虑大且依赖远程 solution 可配置）与 P-C（放弃聊天体验）。

## 关键决策（brainstorm 确认）

1. agent 角色 = **读文件 + 对话里判断/建议，用户点确认入库**（agent 有实质识别参与，但不执行入库）。
2. 操作条位置 = **ChatView 上方一条独立区**。
3. sandbox 目录 = `uploads`；选文件后**自动发**提示消息让 agent 看。
4. **解耦原则**：本地入库只依赖前端手里的 File 字节，与「传 sandbox + agent 建议」解耦——agent 读不了/不可达也不挡保存。

## 既有能力（复用，不改）

- 右栏 `ChatPanel`（`src/layout/ChatPanel.tsx`）：`bootstrapBladeClient` 建 `bladeClient`，建会话拿 `sessionId`，渲染 `<ChatView sessionId/>`。
- 本地 API `src/api/standardDocs.ts`：`uploadStandardDocs(files: File[]) -> UploadResult`（→ `POST /api/standard-docs`）。
- 规则库页 `StandardDocLibrary` 已 10s 轮询，agent/前端入库后自动出现新行。
- Blade SDK（`@blade-hq/agent-kit@0.5.11`，从 `/react` 导入）：
  - `uploadFiles(sessionId, dirPath, files: File[]) -> {uploaded: string[], failed: string[]}`：传文件到会话 sandbox。
  - `useChat(sessionId) -> { messages, isStreaming, send, stop }`；`send(content: MessageContent, mode?, …)` 程序化发消息，与 `<ChatView>` 共享会话 store，发出的消息显示在对话里。
  - `buildMessageContent(text) -> MessageContent`：构造消息内容。

## 架构与组件

| 组件 | 位置 | 职责 | 状态 |
|------|------|------|------|
| 本地入库接口 | 后端 `POST /api/standard-docs` | 写 `file_object`+`standard_doc` | 已存在，**不改** |
| **`RuleDocChatBridge`** | 本仓库 `src/components/RuleDocChatBridge.tsx` | ChatView 上方操作条：选文件→传 sandbox+发提示→用户点确认→本地入库 | **新建** |
| ChatPanel 接入 | `src/layout/ChatPanel.tsx` | 会话就绪时在 `<ChatView>` 上方渲染 `<RuleDocChatBridge sessionId/>` | **小改** |
| 规则库页轮询 | `StandardDocLibrary` | agent/前端入库后自动出现新行 | 已存在，不改 |

## 数据流

```
用户点「选规则文件给 AI 看」→ 选 政策A.pdf
   ├─(a) 前端 state 留住 File 字节
   ├─(b) uploadFiles(sessionId,"uploads",[file]) → 落到 agent sandbox
   └─(c) send(buildMessageContent("我上传了 政策A.pdf,请判断是否规则文档(政策/指南/规范)并建议是否入库"))
              ▼
   远程 agent 读 sandbox 文件 → 在 ChatView 回复识别/建议(供参考)
              ▼
用户点「存入规则库」(决策门)
              ▼
   uploadStandardDocs([file]) → 本地 POST /api/standard-docs → file_object+standard_doc
              ▼
   toast「已存入:政策A(SD-xxxx)」;规则库页 10s 轮询出现新行
```

## 组件设计：`RuleDocChatBridge`

- **Props**：`{ sessionId: string }`（仅在会话就绪时由 ChatPanel 渲染）。
- **State**：`pickedFile: File | null`、`aiBusy: boolean`（传 sandbox/发消息中）、`saving: boolean`。
- **UI**：一条横向操作条——按钮「选规则文件给 AI 看」（隐藏 `<input type=file>`）+ 按钮「存入规则库」（`disabled = !pickedFile || saving`）+ 状态文字。
- **选文件 onPick**：
  1. `setPickedFile(file)`。
  2. best-effort：`await uploadFiles(sessionId, "uploads", [file])` → 成功后 `send(buildMessageContent("我上传了 " + file.name + ",请判断它是不是规则文档(政策/指南/规范),并建议是否存入规则库。"))`。
  3. 失败（sandbox 上传或发消息抛错）→ `toast.warning("传给 AI 失败,但你仍可直接存入规则库")`，**不清 pickedFile、不挡保存**。
- **存入规则库 onSave**：
  1. `setSaving(true)`；`const res = await uploadStandardDocs([pickedFile])`。
  2. `res.uploaded.length` → `toast.success("已存入规则库:" + title + "(" + doc_code + ")")`；`res.failed.length` → `toast.warning(...原因)`。
  3. best-effort：`send(buildMessageContent("已存入规则库:..."))` 让对话反映结果。
  4. 清空 `pickedFile`、`setSaving(false)`。出错 → `toast.error`。
- **通知**：用 `sonner` 的 `toast`（与 `StandardDocLibrary` 一致）。
- 选完文件 `input.value = ""`（允许重选同名文件）。

## 错误处理 / 韧性

- 传 sandbox / 发消息失败 → toast 警告，不挡保存（agent 建议是 best-effort）。
- 本地入库失败 → 复用 `StandardDocLibrary` 同款 toast 处理。
- `uploadStandardDocs` 的 `failed[]` → 逐项 toast 原因（超限等）。
- 决策门 = 用户点「存入规则库」，绝不自动入库。

## 测试

- 新增 `src/components/RuleDocChatBridge.test.tsx`（vitest）：
  - mock `@blade-hq/agent-kit/react`：`uploadFiles`、`useChat`（返回含 `send` 的对象）、`buildMessageContent`。
  - mock `../api/standardDocs`：`uploadStandardDocs`。
  - mock `sonner`：`toast`。
  - 用例：① 选文件 → `uploadFiles` 以 `(sessionId,"uploads",[file])` 调 + `send` 被调且参数含文件名；② 点「存入规则库」→ `uploadStandardDocs([file])` 被调 + 成功 toast；③ **`uploadFiles` reject 时仍能点「存入规则库」并成功**（验证解耦）。
- ChatPanel 集成不回归：现有 `ChatPanel.test.tsx`（4 个）中 mock 掉 `RuleDocChatBridge`（`vi.mock`）或其 SDK hook，避免 `useChat` 真连 socket。
- 后端零改动，后端测试不受影响。全量前端测试在现有 27 基础上新增。

## 范围边界

- **全部在本仓库前端**：`RuleDocChatBridge.tsx` + 测试 + ChatPanel 接入。后端不动。
- **Blade 侧零配置**：无需 skill / shim / solution 控制权 / 自托镜像。
- **唯一假设**：远程 agent 能读 sandbox 文件并给建议；已解耦，读不了也不影响保存。

## 端到端人工验证（需有效 Blade token + 真实 smart 库）

1. `.env` 填有效 `VITE_BLADE_TOKEN`；起后端（`backend/` `uvicorn ... :8000`）+ 前端（`npm run dev`）。
2. 右栏会话就绪 → ChatView 上方出现「规则文件入库」操作条。
3. 点「选规则文件给 AI 看」选 `政策A.pdf` → 对话里 agent 给出识别/建议。
4. 点「存入规则库」→ toast 成功 → 切到规则库页 10s 内出现新行；查 `smart.standard_doc`/`file_object` 有行、磁盘有文件。
5. 反例:断开后端或传 sandbox 失败 → 操作条提示传 AI 失败,但点「存入规则库」仍能保存(或本地入库报错 toast)。

## 验收标准

1. ✅ 操作条在 ChatView 上方;选文件 → 传 sandbox + 自动发消息 → agent 对话内识别/建议。
2. ✅ 点「存入规则库」→ 前端调本地 `POST /api/standard-docs` 入库 → toast + 规则库页轮询出现新行。
3. ✅ 解耦:sandbox 上传/发消息失败不挡本地保存。
4. ✅ 后端零改动、Blade 侧零配置;现有前端 27 测试不回归 + 新增组件测试全绿。

## Out of Scope / YAGNI

- agent 自动触发保存（客户端工具执行，SDK/solution 前提不具备）。
- 公网暴露后端 / 自托 blade-agent（P-A、blade-coa compose 路）。
- 内容提取/结构化（只存原文件）。
- 钩 ChatView 原生附件取字节（改用自有文件选择器，避免碰 SDK 内部）。
- 申请材料经聊天入库;聊天里删除/管理规则文件（仍走规则库页）。
- save_rule_doc 的 shim/skill 部署（已在仓库，待将来具备自托/可达条件再启用）。
