# 形式审查 · 子项目 C：审查执行 — 设计文档

> 日期：2026-06-25　状态：设计已确认（待用户复审 spec）
> 所属：「立项审查 AI 辅助系统」之**形式审查**能力（系统三环节第二环：规则准备✅ → **形式审查⏳** → 专家审查⏳）
> 前置：A（上传识别）✅(HEAD df48185)、B（结构化抽取）✅(HEAD 41df217)。`material_file`/`parse_segment`/五类结构化对象已可落库。

## 1. 背景与定位

形式审查整条流水线水平分 A→B→C→D：A 上传识别✅、B 结构化抽取✅、**C 审查执行（本文）**、D 报告。

`docs/系统提示词.md` 定义形式审查 = 依**硬性(hard)规则**逐条核对申请材料，客观判定，每条输出**四要素**（结论/依据/出处/建议），按维度汇总。映射到 schema：
- **结论** → `round_check.initial_result`（pass/fail/need_review/not_applicable）+ `initial_disposition`（reject/fix/review）
- **依据** → 该规则的 `review_rule_version`（name/logic/dimension）及其依据条款
- **出处** → `finding_evidence`（指向申请材料的 segment / field_code / budget_item）
- **建议** → `round_check.suggestion`

C 落 `round_check`+`finding_evidence`，聚合 `review_round.conclusion`，并支持**人工复核**（`final_result`/`final_disposition` + `check_review_action` 历史 + 乐观锁）。

**驱动架构**：延续 A/B 与规则侧——**Blade agent 做四要素判定**（LLM 逐条），后端提供「物化配置 + 取审查输入 + 幂等写结果 + 人工复核」能力端点（cloudflared 隧道 + X-API-Key 写鉴权 + python shim）。人工复核由前端直接调后端（同 `authHeaders()` 注入 X-API-Key）。

## 2. 本期目标与验收

**目标**：agent 在 Blade 聊天对某已结构化抽取的审查包说"用《XX规则》审查这个申报包" → 经 skill `review-package`：选配置包(规则文件)物化绑批次 → 取审查输入(hard 规则 + 结构化数据 + 段落) → LLM 逐条判四要素 → 写 `round_check`+`finding_evidence` → 聚合 round 结论；前端「形式审查」页能看到该包的审查结论与逐条结果，并支持人工 confirm/overrule。

**验收标准**：
1. `POST /api/packages/{id}/bind-config` 传 config_doc_id → 物化 `config_package`+`config_rule_version`，给该包**新建独立 batch** 并设 `config_id`，返回 `{config_id, rule_count}`。
2. `GET /api/packages/{id}/review-input` → 返回该 config 下 hard 规则 + 该包结构化数据 + 段落。
3. `POST /api/packages/{id}/review` 传 checks → 建/取 round、写 `round_check`(初判)+`finding_evidence`、聚合 `review_round.conclusion`；**重跑替换不累积**。
4. `POST /api/round-checks/{id}/review-action` confirm/overrule → 写 `final_result`+`check_review_action`+自增 `version`；**version 不符 → 409**；之后 round 结论按 effective 重聚合。
5. 前端「形式审查」页（`review-tasks`）：列有 round 的审查包 → 展开看结论 + 逐条 round_check（规则/初判/建议/证据出处）+ 复核按钮可改 final。
6. 后端全量测试绿；前端 vitest 绿 + `npm run build` 零错误。

**非目标（YAGNI，留给 D 或更后）**：审查报告生成（D）；多轮次（round_no 恒为 1）；案例域 `review_case`；`sim_run` 外部核验；config 的规则子集选择（本期物化该规则文件全部规则，hard 子集在判定时筛）；真实登录用户（复核 actor 用 `__system__`）；按维度加权汇总分值（`config_package.dimension_weight`）。

## 3. 规则来源链与 config 物化

规范链：`application_package → batch_id → review_batch.config_id → config_package → config_rule_version → review_rule_version`。

### 3.1 config 物化（复用配置包派生口径）
新增 `materialize_config(db, config_doc_id) -> int`：
- 派生该 `standard_doc` 的 active `rule_version_id` 集（口径同 `routers/config_packages.list_config_packages`：`RegulationClause→ReviewRuleClause→ReviewRuleVersion→ReviewRule(current_version,is_active)`，`standard_doc.is_active`）。
- 幂等 get-or-create `config_package`：`code=f"CFG-{doc_code}"`（按 code 去重，UNIQUE 语义），`project_type_id`/`stage_id`=默认主数据，`name=doc.title`、`version="V1.0"`、`status="published"`。
- 清该 config 旧 `config_rule_version`，重插派生出的全部 rule_version_id。
- 返回 config_id。doc 不存在或无规则 → `ValueError`。

### 3.2 绑批次（每包独立 batch）
新增 `bind_package_config(db, package_id, config_doc_id) -> (config_id, rule_count)`：
- 包不存在 → `LookupError`。
- `materialize_config` 取 config_id。
- **每包独立 batch**：若该包当前 batch 是共享默认 batch（`batch_no="__DEFAULT_BATCH__"`），则克隆其 `project_type_id`/`stage_id`/`created_by` 新建一条 batch（`batch_no=f"BATCH-PKG{package_id}"` 幂等去重）并把 `application_package.batch_id` 改指向它；否则复用该包既有专属 batch。设该 batch 的 `config_id=config_id`。
- `rule_count` = 该 config 下 hard 规则数。
- 单次 commit。

> 取舍：config 绑在 batch 上是 schema 规范。A 的共享占位 batch 会导致跨包串用，故绑定时为该包派生专属 batch，既独立又不破坏既有上传链路。

## 4. 后端能力（`review_execution.py` + `routers/review.py`）

### 4.1 `POST /api/packages/{id}/bind-config`（写，`Depends(require_api_key)`）
body `{config_doc_id:int}` → 调 `bind_package_config` → `{config_id, rule_count}`。`LookupError→404`、`ValueError→422`。

### 4.2 `GET /api/packages/{id}/review-input`（读）
返回 agent 判定所需输入：
```
{
  config_id, package_id,
  rules: [{rule_version_id, rule_code, name, logic, dimension_code, dimension_name, disposition, clause_no, clause_text}],  // 仅 hard
  members:[...], coop_units:[...], budget_items:[...], attachments:[...], fields:[...],   // 复用 B 的 structured 读出
  segments: [{material_file_id, file_name, segments:[{id,page_no,segment_type,content_text}]}]
}
```
包不存在 → 404；batch 未绑 config（config_id 为空）→ 422（提示先 bind-config）。

### 4.3 `POST /api/packages/{id}/review`（写，`Depends(require_api_key)`）
body：
```
{checks:[{rule_version_id, initial_result, initial_disposition?, suggestion?, confidence?, severity?,
          evidence:[{segment_id?, field_code?, budget_item_id?, note?}]}]}
```
逻辑：
1. 包不存在 → 404；包未绑 config → 422。
2. 建/取 `review_round(package_id, round_no=1, conclusion='pending')`。
3. **全量前置校验**（先于任何写）：每 `rule_version_id` ∈ 该 config 的 hard 集（否则 422）；`initial_result`∈白名单（否则 422）；`initial_disposition` 非空时∈{reject,fix,review}；evidence 的 `segment_id`∈本包段落 / `budget_item_id`∈本包预算 / `field_code` 为字符串（否则 422）。
4. **幂等**：删该 round 的 `finding_evidence`（经 round_check 反查）+ `round_check`，重插。`initial_disposition` 缺省取该 rule_version 的 `disposition`。
5. 聚合 `review_round.conclusion`（见 §6）。
6. 返回 `{round_id, round_no, conclusion, checks_written, evidence_written}`。

### 4.4 `GET /api/packages/{id}/review`（读）
返回 `{round:{round_id, round_no, conclusion}, checks:[{round_check_id, rule_version_id, rule_code, name, dimension_code, initial_result, initial_disposition, final_result, final_disposition, effective_result, status, suggestion, confidence, severity, version, evidence:[{segment_id, field_code, budget_item_id, note}]}]}`。包无 round → `{round:null, checks:[]}`。

### 4.5 `POST /api/round-checks/{id}/review-action`（写，`Depends(require_api_key)`）
body `{action, final_result?, final_disposition?, remark?, version}`：
- round_check 不存在 → 404；`version` 与当前不符 → **409**（乐观锁）。
- `action="confirm"`：`final_result=initial_result`、`final_disposition=initial_disposition`、`status="confirmed"`。
- `action="overrule"`：`final_result`/`final_disposition`=给定（校验白名单，否则 422）、`status="overruled"`。
- 写 `check_review_action(round_check_id, actor_id=__system__, action, from_result=有效旧值, to_result, remark)`；`round_check.version += 1`、set `reviewed_by`/`reviewed_at`/`review_remark`。
- 重聚合所在 round 的 `conclusion`。
- 返回更新后的 check（含新 version）。

## 5. 机审落库（round_check + finding_evidence）

- `review_round`：`(package_id, round_no=1)`，初始 `conclusion='pending'`。
- `round_check`：`(round_id, rule_version_id)`（UNIQUE），`initial_result`∈{pending,pass,fail,need_review,not_applicable,error}，`initial_disposition`∈{reject,fix,review}|NULL，`status='open'`，`version=0`，`applied_param_snapshot=NULL`（本期不做参数化规则）。
- `finding_evidence`：每条证据指向 `segment_id`|`field_code`|`budget_item_id` 之一（均可空，至少给一类；`sim_run_id` 恒 NULL）。

## 6. round 结论聚合

按各 check 的 `effective_result = COALESCE(final_result, initial_result)` 与有效处置（final 优先）聚合 `review_round.conclusion`：
1. 任一 effective=`fail` 且有效处置=`reject` → `reject`
2. 否则任一 effective=`fail`（处置=fix 或空）→ `fix`
3. 否则任一 effective∈{`need_review`,`pending`,`error`} → `pending`
4. 否则（全 pass/not_applicable）→ `accept`

机审写入后、每次人工复核后均重算。

## 7. Skill `review-package` + shim

`blade/skills/review-package/`（对齐既有 skill 目录布局：`versions/1.0.0/SKILL.md`+`metadata.json`，shim 走 `shared/helpers/`）。
- 触发：用户对已结构化抽取的审查包表达"审查/形式审查这个申报包"（可指定规则文件）。
- 流程：agent `GET /config-packages` 选 config_doc（按用户意图）→ shim `smart_doc_review_input.py`（POST bind-config + GET review-input，打印 JSON）→ LLM 逐条 hard 规则判四要素（带 evidence 出处）→ shim `smart_doc_review.py`（POST /review，payload.json）→ `GET /packages/{id}/review` 回读 → 忠实回报结论与各计数。
- SKILL.md 含 `initial_result`/`initial_disposition` 白名单与铁律（只有退出码 0 才算成功，不编造）。

## 8. 前端「形式审查」页（填充 `review-tasks` / `ReviewTasksPage`）

复用既有「审查」导航的 `review-tasks` 槽（无需改 `menuConfig`）：
- 列出有 round 的审查包（复用 `listMaterialPackages` 列包 + 每包 `GET /packages/{id}/review` 取结论/明细；无 round 的包显示「未审查」）。
- 展开：round 结论 Tag + `round_check` 表（规则名/维度/初判 Tag/处置/建议/证据出处）。
- 复核交互：每条 confirm / overrule 按钮 → 调 `POST /round-checks/{id}/review-action`（带 version 乐观锁 + `authHeaders()`）→ 刷新；overrule 弹窗选 final_result/disposition + remark。
- 新增 `src/api/review.ts`（`getPackageReview` + `postReviewAction`，写带 `authHeaders()`）+ 组件 + vitest。

## 9. 错误处理

- 包/规则文件/round_check 不存在 → 404；config_doc 无规则 / 未绑 config / 枚举越界 / evidence 不属本包 → 422；乐观锁 version 不符 → 409；无 X-API-Key（启用时）→ 401。
- 空 checks → 正常建 round、写 0 条、conclusion=accept（无 fail）。
- 前端：结果 isError / 复核失败(409 提示刷新) / 空态分别处理。

## 11. 复用与技术债

**复用**：B 的 structured 读出（review-input 内联五类）、`parse_segment` 段落聚合（同 `/packages/{id}/segments`）、config_packages 派生口径、写端点 X-API-Key、前端 `authHeaders()`、skill 目录布局与 shim 模式、antd Table/Tabs/只读展开。

**技术债 / 后续**：
- **重跑机审清掉人工复核**（幂等替换 round_check 会丢 final_*/check_review_action）：MVP 视重跑为全新初判；将来需「仅替换 status='open' 的 check」或版本化轮次。
- config 物化含该规则文件**全部** rule_version，hard 子集在判定时筛（未在 config 层区分 stage 用途）。
- 复核 `actor_id` 用 `__system__` 默认用户（无真实登录/权限）。
- `round_no` 恒 1（无复审轮次）；`applied_param_snapshot` 恒 NULL（无参数化规则）；`config_package.dimension_weight` 未用（无加权汇总）。
- bind-config 改 `application_package.batch_id` 后，A 的占位 declared_project/共享 batch 关系仍为脚手架（同 A/B 技术债）。
