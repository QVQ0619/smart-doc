# 规则结构化（子项目3）设计

> 立项审查 AI 系统「文档识别 + 规则抽取 + 规则结构化」功能的**第三个子项目**。
> 消费子项目2 产出的 `regulation_clause`（带出处），把它升格为带结构化判定字段的
> `review_rule` + `review_rule_version` + `review_rule_clause`（关联回源条款、继承出处）。

## 目标

用户在聊天里让 Blade agent 把某文档**已抽取**的审查条款（`regulation_clause`）结构化为
**审查规则**：agent 经后端读该文档条款、为每条判定 6 个结构化字段（维度 / 判定类型 /
处置 / 绑定类 + 规则名 + 判定逻辑），写入 `review_rule`（身份）+ `review_rule_version`
（不可变版本 V1.0）+ `review_rule_clause`（关联回源 clause）；规则库页展开行用 Tabs
切换「审查规则」「依据条款」两视图，规则视图展示结构化字段并继承条款出处。

## 背景与既有事实

- **子项目2 已交付**（HEAD 9e53eca，且本地已端到端实跑：doc3 识别15段→抽8条款带出处→幂等）：
  `regulation_clause`（`id`/`standard_doc_id`/`doc_code`/`clause_no`/`clause_text`/
  `source_segment_id`→`parse_segment`）。出处链 `clause.source_segment_id` →
  `parse_segment.page_no`+`locator`。
- **开工前小改已完成**（未提交）：`extraction.replace_clauses` 在 doc 不存在时
  `raise ValueError`（fail-fast），后端 54 passed。
- **本子项目涉及的 schema（5.7/5.12/5.13/5.14，DB 预建、权威，不改）**：
  - **`review_dimension`**（5.7，6 行主数据，**当前为空，需 seed**）：`id`、`code`(VARCHAR16,
    UNIQUE)、`name`(VARCHAR16)。六维：完整性/规范性/合规性/一致性/合理性/真实性。
  - **`review_rule`**（5.12，规则身份）：`id`、`rule_code`(VARCHAR24, **UNIQUE NOT NULL**)、
    `current_version_id`(FK→review_rule_version，可空)、`is_active`、`created_at`、`updated_at`。
    **无 standard_doc_id 字段**。
  - **`review_rule_version`**（5.13，不可变版本）：`id`、`rule_id`(FK→review_rule, NOT NULL)、
    `version`(VARCHAR16, NOT NULL)、`dimension_id`(FK→review_dimension, NOT NULL)、
    `name`(VARCHAR160, NOT NULL)、`logic`(TEXT 可空)、`decision_type`(VARCHAR8, NOT NULL,
    CHECK `IN ('hard','verify','soft')`)、`disposition`(VARCHAR8, NOT NULL, CHECK
    `IN ('reject','fix','review')`)、`binding_class`(VARCHAR16, NOT NULL, CHECK
    `IN ('common','parameterized','specific')`)，外加 embedding 列（本子项目留空）。
    **UNIQUE(rule_id, version)**。
  - **`review_rule_clause`**（5.14，规则版本×依据条款）：`id`、`rule_version_id`(FK→
    review_rule_version, NOT NULL)、`clause_id`(FK→regulation_clause, NOT NULL)、`note`(可空)。
    **UNIQUE(rule_version_id, clause_id)**。
- **⚠️ 循环外键**：`review_rule.current_version_id` → `review_rule_version.id` 与
  `review_rule_version.rule_id` → `review_rule.id` 互指。schema 建表时用
  `SET FOREIGN_KEY_CHECKS=0` 解决。写入顺序须：插 rule(current_version_id=NULL) →
  flush 取 rule.id → 插 version(rule_id=rule.id) → flush 取 version.id → 回填
  rule.current_version_id → 单次 commit。
- **⚠️ review_rule 无 doc 归属字段**：「某文档的 review_rule」只能经
  `review_rule_clause.clause_id` → `regulation_clause.standard_doc_id` 反查。按文档 replace
  与 GET-by-doc 都依赖此反查链。
- **会话推送 skill 模式**：`src/blade/sessionSkill.ts` 用 `partnerSkillApi.uploadSessionSkill`
  推本地 skill 到 agent 沙箱。技能名须匹配 `^[a-z0-9-]+/[a-z0-9-]+$`。现已推
  `local/save-rule-doc`、`local/extract-review-rules`。`api_base.txt` 由 `VITE_SMART_DOC_API` 注入。
- **测试基线**：后端连真实 MySQL `smart_test`（conftest 用 schema SQL 建库；清理需先删子表，
  顺序须含新表 review_rule_clause → review_rule_version → review_rule）。

## 范围

### 本子项目做
1. `review_dimension` 6 行幂等 seed（后端 startup ensure）。
2. 后端服务 `structuring.replace_rules` + 路由 `POST/GET /standard-docs/{id}/rules`（并入
   现有 `clauses.py` router，复用 `_require_doc`）。
3. skill③ `local/structure-review-rules`（会话推送）+ 两个 shim（读条款 + 写规则）。
4. `sessionSkill.ts` 扩展为推送三个 skill。
5. 前端规则库展开行改为 Tabs：「审查规则」（新）+「依据条款」（复用现有 ClauseList）。

### 本子项目不做（明确推迟）
- **`rule_project` / `rule_stage` / `rule_param`** 关联表（适用项目类型/阶段/阈值参数）——需
  `project_type` 主数据与阈值抽取逻辑，**留后续**。`binding_class` 本轮仅作 agent 标注字段。
- **版本演进 / 不可变历史**：本轮按文档 replace（删旧重建 V1.0），不追加版本、不保留历史。
  （`review_rule_version` 的不可变审计语义留待 config_package 阶段真正需要时。）
- **embedding/向量化**：review_rule_version 的 embedding 列留 NULL。
- **schema 变更**：不新增任何列/表（DB 预建、权威）。「某文档是否已结构化」由该文档经反查链
  是否存在 review_rule 判断，不加状态列。
- **结构化质量保证 / 完备性**：判定是 agent 行为，不保证判全/判准；靠铁律确保只如实回报实际
  入库条数、字段取自真实条款。

## 架构与数据流

```
聊天「把《申请规定》的审查条款结构化为规则」
  └─ skill③ structure-review-rules（会话推送，同 extract-review-rules 机制）
       1. 解析目标 doc_id：smart_doc_list_clauses 接受 doc_id/doc_code/标题
          （纯数字→id；SD-开头→doc_code；否则→标题子串；多条→退出码7列候选；0条→8）
       2. GET /api/standard-docs/{id}/clauses → 该文档全部 regulation_clause
          (id/clause_no/clause_text/出处)
       3. ★agent 判定★：为每条 clause 产出
          {source_clause_id, dimension_code, name, logic, decision_type, disposition, binding_class}
       4. agent 写 JSON 文件 → shim POST /api/standard-docs/{id}/rules
       5. 后端按文档幂等替换：删该文档旧 review_rule(+version+rule_clause) → 重建
          （循环 FK 回填 current_version_id；非法 dimension_code/clause_id/枚举 → 跳过计 skipped）
       6. agent 回报「已结构化 N 条规则」（铁律：只转述 shim 实返 inserted）
  └─ 前端规则库行展开 → Tabs[审查规则] → GET /api/standard-docs/{id}/rules → 列结构化字段+出处
```

## 组件

### 主数据 seed `backend/app/dimensions.py`（新建）
```python
DIMENSIONS = [
    ("completeness", "完整性"), ("normativeness", "规范性"), ("compliance", "合规性"),
    ("consistency", "一致性"), ("rationality", "合理性"), ("authenticity", "真实性"),
]
def ensure_dimensions(db: Session) -> None:
    """幂等：按 code upsert 6 行 review_dimension（已存在不动，缺失补插）。单次 commit。"""
```
- 在 `app/main.py` 的 startup 事件调用 `ensure_dimensions`（用一个独立 Session）。
- agent 用 `code` 指定维度；后端解析 code→dimension_id。

### 后端服务 `backend/app/structuring.py`（新建）
```python
def replace_rules(db: Session, doc_id: int, rules: list[RuleIn]) -> RuleWriteResult:
    """按文档幂等替换该文档的 review_rule（1:1 升格）。
    1. 合法 clause 集合 = {id: regulation_clause where standard_doc_id=doc_id}。
    2. 删旧：经 review_rule_clause.clause_id ∈ 该文档 clause 反查出 rule_version_id →
       rule_id；按子表→父表顺序删 review_rule_clause、review_rule_version、review_rule。
       （删 review_rule 前先把 current_version_id 置 NULL，避免循环 FK 阻塞删除。）
    3. dim_map = {code: id} 来自 review_dimension。
    4. 逐条 rule 校验后建：
       - source_clause_id 不在合法 clause 集合 → skipped += 1，跳过。
       - dimension_code 不在 dim_map → skipped += 1，跳过。
       - decision_type/disposition/binding_class 不在各自 CHECK 白名单 → skipped += 1，跳过。
       - 合法：插 review_rule(rule_code=gen(), current_version_id=NULL) → flush →
         插 review_rule_version(rule_id, version='V1.0', dimension_id, name, logic,
         decision_type, disposition, binding_class) → flush →
         回填 rule.current_version_id=version.id →
         插 review_rule_clause(rule_version_id=version.id, clause_id=source_clause_id) →
         inserted += 1。
    5. 单次 commit。返回 RuleWriteResult(inserted, skipped)。"""
```
- `rule_code` 生成：`"RULE-" + uuid4().hex[:12]`（≤24 字符，UNIQUE 概率足够；replace 模式每次重建）。
- `name` 必填（NOT NULL）；agent 未给或空 → skipped（不静默写空名）。`logic` 可空。
- 调用方（路由）先保证 doc 存在（404）；本函数假定 doc 存在（不存在时同 extraction 走 fail-fast：
  doc 无 clause → 合法集合为空 → 所有 rule 因 source_clause_id 非法被 skipped，inserted=0）。

### 后端路由 `backend/app/routers/clauses.py`（追加，复用 `_require_doc`）
- **`POST /api/standard-docs/{doc_id}/rules`** → body `RuleBatchIn` → 调 `replace_rules` →
  `RuleWriteResult`；doc 不存在/软删→404；空 rules → inserted=0/skipped=0（不报错）。
- **`GET /api/standard-docs/{doc_id}/rules`** → `list[RuleOut]`，join 链
  `review_rule`（is_active=1）→ `current_version`(review_rule_version) →
  `review_rule_clause` → `regulation_clause`（限 standard_doc_id=doc_id）→ LEFT JOIN
  `parse_segment` 取出处；按 review_rule.id 排序；doc 不存在/软删→404。
  （1:1 升格：每条 rule 经其 current_version 关联唯一一条本文档 clause；用该 clause 取出处。）

### 后端 schemas `backend/app/schemas.py`（追加）
```python
class RuleIn(BaseModel):
    source_clause_id: int
    dimension_code: str
    name: str
    logic: Optional[str] = None
    decision_type: str        # hard|verify|soft
    disposition: str          # reject|fix|review
    binding_class: str        # common|parameterized|specific

class RuleBatchIn(BaseModel):
    rules: list[RuleIn]

class RuleWriteResult(BaseModel):
    inserted: int
    skipped: int              # 非法 clause/维度/枚举/空名 被跳过的条数

class RuleOut(BaseModel):
    id: int                   # review_rule.id
    rule_code: str
    version: str
    name: str
    logic: Optional[str]
    dimension_code: str
    dimension_name: str
    decision_type: str
    disposition: str
    binding_class: str
    source_clause_id: Optional[int]
    clause_no: Optional[str]
    clause_text: Optional[str]
    page_no: Optional[int]    # 来自 join parse_segment（经 clause 继承）
    locator: Optional[Any]
```

## 出处与校验
- 出处渲染：复用子项目2 `clauseProvenance`（page_no 非空→「第X页第Y段」；docx 无页码→「第Y段」）。
- POST 校验（后端，逐条不整批失败）：`source_clause_id` ∉ 本文档 clause 集合 / `dimension_code`
  ∉ 6 维 / `decision_type|disposition|binding_class` ∉ CHECK 白名单 / `name` 空 → 该条 skipped。
- `rule_code` 后端生成、`version='V1.0'` 后端固定、`dimension_id` 后端由 code 解析、出处经 clause
  继承——agent 不提供这些。

## 错误处理
- doc 不存在/软删（is_active=False）→ POST/GET rules 均 404。
- POST 空 rules → inserted=0、skipped=0（不报错）。
- 目标文档无 regulation_clause（未抽取）→ GET clauses 返回空；SKILL.md 指示 agent 先提示「该文档
  尚未抽取条款，请先抽取」，不硬升格、不编造。
- 目标文档解析：标题命中多条→ smart_doc_list_clauses 退出码 7 + 列候选，agent 请用户用 doc_code
  指明；0 条→退出码 8。
- shim 网络/非2xx → 退出码区分，SKILL.md 据码如实回报，绝不谎报成功。

## 测试策略（后端连 MySQL smart_test + TestClient；前端 vitest）
- **`ensure_dimensions` 测**：空库调用 → 6 行；再调一次 → 仍 6 行（幂等，不重复插）。
- **`replace_rules` 服务测**：构造 doc + parse_segment + regulation_clause → 传 rules（含一个非法
  source_clause_id、一个非法 dimension_code、一个非法 decision_type、一个空 name）→ 断言 inserted/
  skipped 计数正确、review_rule/version/rule_clause 行数一致、current_version_id 已回填且指向正确
  version、rule_clause 关联回正确 clause；**幂等**：再调一次断言替换不累加；**按文档隔离**：另一
  文档的 rule 不被本次 replace 删除。
- **接口测**：POST rules 写入并返回计数；GET rules join 出结构化字段 + 维度 name + 条号/条文/出处
  （page_no/locator 经 clause→parse_segment）；doc 不存在→404；软删→404；空 rules→inserted=0；
  未结构化文档 GET rules→空列表。
- **shim 测**（stub HTTPServer）：`smart_doc_list_clauses` GET 打印 doc_id + 条款 JSON、退出码
  （多条→7、0条→8、缺 env→6、非2xx→4）；`smart_doc_rules` 读 JSON POST、打印 inserted/skipped、
  退出码（缺文件→2、缺 env→6、非2xx→4）。
- **前端测**：`listRules` 发对 GET；展开行 Tabs 渲染（规则 Tab 列出结构化字段+出处、条款 Tab 复用
  ClauseList）；规则空显示提示。全量 `npm test` 通过、`npm run build` 零类型错误。
- conftest 清理顺序追加新表：review_rule_clause → review_rule_version → review_rule（在
  regulation_clause 之前删，因 rule_clause FK→clause）。

## 验收标准
1. 后端启动后 `review_dimension` 恰有 6 行（再次启动不重复）。
2. 聊天里对一个已抽取条款的文档说「把审查条款结构化为规则」，agent 经 skill③ GET 条款、逐条判定后
   POST，`review_rule`/`review_rule_version`/`review_rule_clause` 出现该文档多条：rule_code 唯一、
   version=V1.0、current_version_id 已回填、dimension/decision_type/disposition/binding_class 合法、
   rule_clause 关联回本文档 clause。
3. 重新结构化同一文档 → 规则被替换而非累加（幂等）；另一文档的规则不受影响。
4. 含非法 source_clause_id / 维度 / 枚举 / 空 name 的提交 → 该条不入库、响应 skipped≥1，agent 如实回报。
5. 规则库页展开某文档行 → Tabs「审查规则」列出结构化字段 + 出处；「依据条款」仍展示 clause；
   未结构化的文档规则 Tab 显示提示。
6. agent 回报条数与后端实际 inserted 一致（铁律，无夸大/编造）；未抽条款时提示先抽取而非硬升格。
7. 后端全量 pytest 通过；前端 npm test 全绿、npm run build 零类型错误。
