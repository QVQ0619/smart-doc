# 审查规则与依据条款的修改 — 设计文档

**日期:** 2026-06-23
**目标:** 让 agent 抽取得到的「审查规则」与「依据条款」支持人工校正——在前端规则库页直接编辑字段、删除单条。

## 背景

规则准备链路(上传→识别→抽取→结构化)已实现,agent 抽取的结果入库后,前端规则库
展开行的「审查规则 / 依据条款」两个 Tab **目前纯只读**。当抽取有误(条号错、条文有
误、维度/判定判错)时,用户只能在聊天里整篇重抽,无法精确校正单条。本功能补上"人工
校正"这一层。

## 数据现状(只读已实现)

- **依据条款** = `regulation_clause`:`clause_no`(条号,NOT NULL)、`clause_text`(条文)、
  `source_segment_id`(出处段落)、`standard_doc_id`(归属文档)、`doc_code`。
- **审查规则** = `review_rule` + `review_rule_version` + `review_rule_clause`,**1:1、单版本 V1.0**:
  - `review_rule`:`rule_code`、`current_version_id`、`is_active`、`updated_at`。
  - `review_rule_version`:`name`、`logic`、`dimension_id`、`decision_type`、`disposition`、`binding_class`。
  - `review_rule_clause`:`rule_version_id` ↔ `clause_id`(把规则关联到依据条款,出处由此继承)。
- 归属反查:规则经 `review_rule_clause→regulation_clause.standard_doc_id` 反查所属文档。
- 已有写端点均为"整篇替换式":`POST .../extract-rules`、`.../clauses`、`.../rules`,带 `X-API-Key`。
- 枚举白名单(已在 `structuring.py`):`_DECISION={hard,verify,soft}`、
  `_DISPOSITION={reject,fix,review}`、`_BINDING={common,parameterized,specific}`。维度 6 行 `review_dimension`。

## 决策(已与用户确认)

1. **入口**:前端规则库页直接编辑(弹窗表单),不走聊天 agent。
2. **可改字段**:条款 = `clause_no`/`clause_text`;规则 = `name`/`logic`/`dimension`/`decision_type`/`disposition`/`binding_class`。
   不含"改出处段落""重新关联条款"。
3. **版本语义**:**原地改当前版本**——直接覆盖 `review_rule_version` 字段,不新建版本、不留历史。
   (规则仍在准备/校正阶段;审查运行域 `round_check` 等快照引用尚未启用,原地改安全。)
4. **删除**:支持删单条(规则、条款),级联语义见下。

## 后端设计

### 端点(新增于 `backend/app/routers/clauses.py`,均 `dependencies=[Depends(require_api_key)]`)

| 方法 | 路径 | 请求体 | 响应 |
|---|---|---|---|
| `PATCH` | `/standard-docs/{doc_id}/clauses/{clause_id}` | `ClauseUpdateIn` | `ClauseOut` |
| `DELETE`| `/standard-docs/{doc_id}/clauses/{clause_id}` | — | `204` |
| `PATCH` | `/standard-docs/{doc_id}/rules/{rule_id}` | `RuleUpdateIn` | `RuleOut` |
| `DELETE`| `/standard-docs/{doc_id}/rules/{rule_id}` | — | `204` |

`doc_id` 用于**校验归属**:目标条款/规则必须属于该文档,否则 `404`,防止跨文档误改。
`_require_doc` 守卫复用现有写法。

### Schema(`backend/app/schemas.py`)

```python
class ClauseUpdateIn(BaseModel):
    clause_no: str            # NOT NULL,非空(strip 后)
    clause_text: Optional[str] = None

class RuleUpdateIn(BaseModel):
    name: str                 # 非空
    logic: Optional[str] = None
    dimension_code: str       # 必须在 review_dimension
    decision_type: str        # 白名单
    disposition: str          # 白名单
    binding_class: str        # 白名单
```

`ClauseOut`/`RuleOut` 复用现有(返回更新后的行,含出处)。

### 业务逻辑

**改条款** — `extraction.update_clause(db, doc_id, clause_id, data) -> RegulationClause`:
- 取 `RegulationClause`,校验 `standard_doc_id == doc_id`,否则 `ValueError`(路由转 404)。
- `clause_no` strip 后为空 → `ValueError`(路由转 422)。
- 改 `clause_no`/`clause_text`,commit。返回行(路由再 join 段落出处组 `ClauseOut`)。

**删条款** — `extraction.delete_clause(db, doc_id, clause_id) -> None`:
- 校验归属。
- 查 `review_rule_clause.clause_id == clause_id` 关联的 `rule_version_id` → 反查 `rule_id`;
  **若有,连带删除该规则全套**(断 `current_version_id` → 删 `review_rule_clause` → `version` → `rule`),
  避免规则失去依据后在 `list_rules`(inner join)里变成查不出的"幽灵"。
- 再删 `regulation_clause`。单次 commit。

**改规则** — `structuring.update_rule(db, doc_id, rule_id, data) -> int`(返回 rule_id):
- 取 `ReviewRule`,经 `current_version_id` 取当前 `ReviewRuleVersion`。
- 校验该规则经 `review_rule_clause→clause` 反查归属 `== doc_id`,否则 404。
- 校验 `dimension_code` 在 `dim_map`、三枚举在白名单、`name` 非空,否则 422。
- **原地更新** `version` 的 `name/logic/dimension_id/decision_type/disposition/binding_class`;
  置 `review_rule.updated_at = datetime.now()`(Python 侧赋值,避免 update values 里拼 SQL 函数)。commit。

**删规则** — `structuring.delete_rule(db, doc_id, rule_id) -> None`:
- 校验归属。断 `current_version_id` → 删 `review_rule_clause`/`version`/`rule` 三表。**条款保留**。
  (复用 `delete_rules_for_doc` 的级联写法,缩到单条 rule_id。)

## 前端设计

### `src/api/standardDocs.ts`

```ts
export interface ClauseUpdate { clause_no: string; clause_text: string | null }
export interface RuleUpdate {
  name: string; logic: string | null; dimension_code: string;
  decision_type: string; disposition: string; binding_class: string;
}
updateClause(docId, clauseId, body): Promise<Clause>   // PATCH,authHeaders()
deleteClause(docId, clauseId): Promise<void>            // DELETE
updateRule(docId, ruleId, body): Promise<Rule>         // PATCH
deleteRule(docId, ruleId): Promise<void>               // DELETE
```

### `src/components/StandardDocLibrary.tsx`

- `CLAUSE_COLS` / `RULE_COLS` 各加一列"操作":**编辑**(打开 Modal)、**删除**(`Popconfirm`)。
- **条款编辑 Modal**:`Form` 含条号(`Input`,必填)、条文(`Input.TextArea`)。
- **规则编辑 Modal**:`Form` 含规则名(必填)、判定逻辑(`TextArea`)、维度/判定/处置/绑定(`Select`,
  选项分别来自 6 维 + 现有 `DECISION_LABEL`/`DISPOSITION_LABEL`/`BINDING_LABEL` 反推 value→中文)。
  - 维度 Select 选项:新增常量 `DIMENSION_LABEL`(同 `DECISION_LABEL` 风格),映射 6 个 `dimension_code`
    (completeness=完整性/normativeness=规范性/compliance=合规性/consistency=一致性/rationality=合理性/authenticity=真实性)
    →中文标签;Select 的 value 用 code,当前值取 `Rule.dimension_code`。
- 因 `ClauseList`/`RuleList` 当前是无状态子组件,需把"打开编辑弹窗"的状态与 mutation 提到
  各自子组件内(用 `useMutation` + 局部 `useState` 管理弹窗开关与当前编辑行),保持组件聚焦。
- 成功后 `toast.success` + `queryClient.invalidateQueries(["clauses", docId])` / `["rules", docId]`。

## 测试

### 后端(`backend/tests/`,连真实 `smart_test`)

- `test_edit_clause.py`:改条款字段成功;空 `clause_no` → 422;跨文档 clause_id → 404。
- `test_edit_rule.py`:原地改规则字段成功(版本号仍 V1.0、id 不变);枚举非法 → 422;非法维度 → 422;跨文档 → 404。
- `test_delete.py`:删规则后条款仍在、规则查不到;删条款后连带规则消失;删不存在/跨文档 → 404。
- 鉴权:无 `X-API-Key` 时(配了 key 的环境)→ 401。

### 前端(`src/components/StandardDocLibrary.test.tsx`)

- 条款行点编辑 → 弹窗出现 → 改条文提交 → 调 `updateClause(docId, id, {...})`。
- 规则行点编辑 → 改维度/判定提交 → 调 `updateRule(...)`。
- 点删除 → 确认 → 调 `deleteClause`/`deleteRule`。
