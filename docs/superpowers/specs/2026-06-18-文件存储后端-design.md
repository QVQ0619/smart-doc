# 立项审查系统 — 文件存储后端（MySQL）设计

- 日期：2026-06-18
- 阶段：第二阶段（新增后端 + 文件上传存储）
- 目标：为前端项目新增 FastAPI 后端，接入本地 MySQL（库 `smart`，用户/密码均 `root`），用于存储上传的文件；规则文件与申请材料分类隔离。

## 1. 背景与前提

- 现状：项目第一阶段是纯前端 SPA（Vite + React 19，:3200），无后端。详见 `2026-06-18-立项审查前端骨架-design.md`。
- 关键前提：**浏览器无法直连 MySQL**，必须有后端服务做中间层。本阶段新增该后端。
- 环境（已确认）：Python 3.13、pip 24.2、MySQL Server 8.4（本地）。
- `smart` 数据库需预先存在：`CREATE DATABASE smart CHARACTER SET utf8mb4;`（表由应用启动自动创建）。

## 2. 架构总览

FastAPI 后端（:8000）作为前端与 MySQL 之间的中间层。前端通过 Vite dev proxy 把 `/api/*` 转发到后端；后端把上传文件落到本地磁盘、把元数据写入 MySQL `smart.documents`。

```
浏览器(前端 :3200) ──/api──▶ Vite proxy ──▶ FastAPI(:8000) ──┬─▶ 磁盘 storage/uploads/<category>/<uuid>.<ext>
                                                          └─▶ MySQL smart.documents (元数据行)
```

## 4. 数据模型（MySQL `smart.documents`）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | BIGINT PK AUTO_INCREMENT | 主键 |
| `category` | VARCHAR(32) NOT NULL | `rule`（规则文件）/ `application`（申请材料） |
| `original_name` | VARCHAR(255) NOT NULL | 上传时的原始文件名 |
| `stored_name` | VARCHAR(255) NOT NULL | 落盘文件名（`<uuid>.<ext>`） |
| `rel_path` | VARCHAR(512) NOT NULL | 相对存储根的路径（含类别子目录） |
| `size_bytes` | BIGINT NOT NULL | 文件大小（字节） |
| `mime_type` | VARCHAR(128) NOT NULL | MIME 类型（缺失时 `application/octet-stream`） |
| `sha256` | CHAR(64) NOT NULL | 内容哈希（完整性/排查用，不做去重） |
| `created_at` | DATETIME NOT NULL | 上传时间（UTC） |

- 表由应用启动时 `Base.metadata.create_all(engine)` 自动创建；本期不引入 Alembic 迁移（YAGNI）。
- `category` 合法值集合在代码中以常量/枚举约束：`{"rule", "application"}`。

## 5. 后端接口（REST，挂在 `/api`）

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| `POST` | `/api/documents` | multipart 上传（多文件）；表单字段 `category` + `files`（一个或多个） |
| `GET` | `/api/documents?category=<rule\|application>` | 按类别列出元数据（按 `created_at` 倒序） |
| `GET` | `/api/documents/{id}/download` | 流式下载原文件（`Content-Disposition` 带原始文件名） |
| `DELETE` | `/api/documents/{id}` | 删除元数据行 + 磁盘文件 |

### 行为细节
- **上传**：
  - `category` 必填且必须 ∈ `{rule, application}`，否则整请求返回 400。
  - `files` 支持一个或多个。**逐文件独立处理**：每个文件按类别落到对应子目录（`storage/uploads/<category>/`），文件名用 `<uuid4>.<原扩展名>`，计算 sha256，写入一行。
  - 单文件最大 **50MB**（可配 `SMART_MAX_UPLOAD_MB`）；超限的文件记入 `failed`（reason 说明超限），不落盘、不入库，不影响同批其它文件。
  - 单文件落盘成功但入库失败时，回滚该行并删除已落盘文件（避免孤儿文件），该文件记入 `failed`。
  - 返回 `{ "uploaded": [DocumentOut, ...], "failed": [{ "name": str, "reason": str }, ...] }`，HTTP 200（即使部分失败；全部失败也是 200 但 `uploaded` 为空——由前端据 `failed` 提示）。
- **列表**：`category` 为必填查询参数，校验合法值（非法返回 400）；只返回该类别文件。
- **下载**：id 不存在或磁盘文件缺失返回 404。
- **删除**：先删 DB 行再删磁盘文件（磁盘文件已不存在则忽略）；id 不存在返回 404。

### 出参模型 `DocumentOut`
`{ id, category, original_name, size_bytes, mime_type, sha256, created_at }`（不暴露 `stored_name`/`rel_path` 等内部落盘细节）。

## 6. 文件存储（`storage.py`）

- 存储根由 `SMART_STORAGE_DIR` 指定（默认 `./storage/uploads`，相对后端工作目录）。
- 按类别分子目录：`<root>/rule/`、`<root>/application/`，启动或写入时确保目录存在。
- 落盘文件名 `<uuid4>.<ext>`（`ext` 取自原始文件名，无扩展名则空）；流式写入并累加 sha256，写完返回 `(stored_name, rel_path, size_bytes, sha256)`。
- 提供 `delete(rel_path)`：删除磁盘文件，不存在则静默。
- `storage/` 目录加入 `.gitignore`。

## 7. 前端对接

- `src/api/documents.ts`：
  - `listDocuments(category)` → `GET /api/documents?category=`
  - `uploadDocuments(category, files: File[])` → `POST /api/documents`（FormData，多文件）→ `{uploaded, failed}`
  - `downloadDocumentUrl(id)` → 返回 `/api/documents/{id}/download`（浏览器直接打开/下载）
  - `deleteDocument(id)` → `DELETE`
  - 用 fetch；列表与变更用已装好的 **@tanstack/react-query**。
- `src/components/DocumentManager.tsx`（可复用）：props `{ category: "rule" | "application"; title: string }`：
  - antd `Upload`（`multiple`，`beforeUpload` 拦截改为受控提交，或用 `customRequest`）一次选多个文件 → 调 `uploadDocuments` → 据 `{uploaded, failed}` 用 antd `message` 提示「成功 N 个 / 失败 M 个（原因）」→ 失效并刷新 react-query 列表。
  - antd `Table` 列：文件名 / 大小（人类可读）/ 类型 / 上传时间 / 操作（下载、删除）。删除前 `Popconfirm` 二次确认。
- `RuleLibraryPage` → `<DocumentManager category="rule" title="规则文件" />`，外层仍用 `PageScaffold`。
- `DocArchivePage` → `<DocumentManager category="application" title="申请材料" />`。
- `vite.config.ts` 增加：`server.proxy = { "/api": { target: "http://localhost:8000", changeOrigin: true } }`。

## 8. 配置

`backend/.env.example`：
```
SMART_DATABASE_URL=mysql+pymysql://root:root@localhost:3306/smart
SMART_STORAGE_DIR=./storage/uploads
SMART_MAX_UPLOAD_MB=50
SMART_CORS_ORIGINS=http://localhost:3200
```
- 后端默认值即上表（`config.py` 用 pydantic-settings 读取，缺省回退到默认）。
- `backend/requirements.txt`：`fastapi`、`uvicorn[standard]`、`sqlalchemy>=2.0`、`pymysql`、`python-multipart`、`pydantic-settings`；测试：`pytest`、`httpx`。
- CORS：`SMART_CORS_ORIGINS` 默认允许前端 `http://localhost:3200`（dev proxy 下通常不触发 CORS，作兜底）。

不做：转发文件到 Blade 会话工作区、用户/权限/鉴权、断点续传、文件去重、Alembic 迁移、对象存储、生产部署/容器化、除 rule/application 外的更多类别。

## 11. 验收标准

1. `smart` 库存在时，启动后端自动建 `documents` 表，无报错。
2. 前端「规则库」页可一次选多个文件上传为规则文件，「审查文档库」页可上传申请材料；各页列表立刻出现自己类别的文件（名/大小/类型/时间），两类互不混淆。
3. 批量上传中个别文件失败（如超限）有明确提示，成功的不受影响。
4. 点下载得到与原文件一致的内容；点删除后列表消失、磁盘文件与 DB 行均删除。
5. 上传 >50MB 文件被拒并提示。
6. 后端 pytest 全绿；前端 vitest 全绿；现有 16 个前端测试不回归。

## 12. 风险与注意

- `smart` 数据库须先手动创建（应用只建表不建库）；root/root 仅适用于本地开发，勿用于生产。
- SQLite 测试与 MySQL 生产存在理论差异（本期 schema 为通用列，差异可忽略；真实 MySQL 行为以人工验证为准）。
- 落盘目录 `storage/` 不入版本库；删除采用「先删行再删文件」，磁盘文件缺失时容错。
- 大文件（接近 50MB）上传为内存/磁盘流式写入，避免一次性读入内存。
