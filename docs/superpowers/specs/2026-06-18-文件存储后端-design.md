# 立项审查系统 — 文件存储后端设计（v2 · 对齐真实 schema）

- 日期：2026-06-18（v2 修订）
- 阶段：第二阶段（新增后端 + 规则文件上传存储）
- 重大修订：放弃 v1 的自创 `documents` 表；改为对齐已部署的 `smart` 库权威 57 表 schema。本期只做**规则文件**（`standard_doc` + `file_object`）；申请材料（`material_file`）因依赖申报批次/项目流程，延后。

## 1. 背景与关键发现

- 真实 MySQL `smart` 库**已部署完整 57 表**（权威来源：`docs/file/审查系统-schema_mysql.sql` 建表脚本、`docs/file/models.py` SQLModel 实体、`docs/file/数据库表说明.md` 说明）。
- 文件存储在该 schema 中有专门归属：
  - `file_object`：通用文件登记（对象存储桥）：`bucket / object_key(唯一) / file_name / mime_type / size_bytes / content_hash / sensitivity / uploaded_by / created_at / deleted_at(软删)`。
  - `standard_doc`（规则文件）：`doc_code(唯一) / title / issuer / pub_year / version / file_id→file_object / is_active / created_at / recognition_status`。
  - `material_file`（申请材料）：依赖 `application_package`(→`review_batch`+`declared_project`) + `secrecy_level` + `material_category`。
- 因此 v1 的独立 `documents` 表与该 schema 冲突（会产生第 58 张游离表，造成"存储混乱"），废弃。

## 2. 本期范围

### 做
- 后端用**真实 schema 的 SQLModel 实体**（`docs/file/models.py`），连真实 `smart` 库（不建表，表已由 SQL 脚本建好）。
- **规则文件**上传/列表/下载/删除，落地到 `file_object` + `standard_doc`，二进制存本地磁盘。
- 前端「规则库」页提供规则文件的上传（多文件）+ 列表 + 下载 + 删除。

### 不做（YAGNI）
- 申请材料 `material_file`（需 `application_package`/`review_batch`/`declared_project` 申报流程，延后）。
- 对象存储（MinIO/OSS）、鉴权、规则条款抽取（recognition）、Alembic 迁移。

## 3. 架构

FastAPI(:8000) 连真实 MySQL `smart`；前端(:3200)经 Vite dev proxy 调 `/api`。规则文件二进制存本地磁盘，元数据写 `file_object`+`standard_doc`。

```
浏览器(:3200) ──/api──▶ Vite proxy ──▶ FastAPI(:8000)
   ├─▶ 磁盘 backend/storage/standard_doc/<uuid>.<ext>
   └─▶ MySQL smart: file_object(bucket=local, object_key=...) + standard_doc(doc_code, title, file_id)
```

## 4. ORM 与数据库连接

- ORM 层直接采用 `docs/file/models.py`（SQLModel 实体，56 表）整文件，复制为 `backend/app/models.py`，**不改动其定义**。
- 连接：`mysql+pymysql://root:root@localhost:3306/smart`。App **不执行 `create_all`**（表由 `审查系统-schema_mysql.sql` 维护）。
- 会话依赖 `get_session()` 产出 `sqlmodel.Session`。

## 5. 涉及的表（本期）

| 表 | 用途 | 本期写入的列 |
| --- | --- | --- |
| `file_object` | 文件二进制登记 | `bucket="local"`、`object_key`、`file_name`、`mime_type`、`size_bytes`、`content_hash(sha256)`、`sensitivity="内部"`；`uploaded_by` 留空 |
| `standard_doc` | 规则文件元数据 | `doc_code`(自动生成唯一 `SD-<12hex>`)、`title`(默认=文件名去扩展名)、`version="V1.0"`、`file_id→file_object`、`is_active=1` |

软删除：删除时 `standard_doc.is_active=0` + `file_object.deleted_at=now()`，磁盘文件保留（尊重审计/软删设计）。

## 6. 接口（REST，挂 `/api`）

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| `POST` | `/api/standard-docs` | multipart 多文件上传规则文件；字段 `files`（一个或多个） |
| `GET` | `/api/standard-docs` | 列出在用规则文件（`is_active=1` 且 `file_object.deleted_at IS NULL`），按 `created_at` 倒序 |
| `GET` | `/api/standard-docs/{id}/download` | 流式下载（`Content-Disposition` 用 `file_object.file_name`） |
| `DELETE` | `/api/standard-docs/{id}` | 软删除（见 §5） |

### 上传行为
- `files` 一个或多个；**逐文件独立处理**：落盘 `backend/storage/standard_doc/<uuid4hex>.<ext>` → 建 `file_object` → 建 `standard_doc`（`title=文件名去扩展名`、`doc_code=SD-<uuid4hex前12>`）。
- 单文件最大 **50MB**（`SMART_MAX_UPLOAD_MB`）；超限/落盘失败/入库失败 → 记入 `failed`，回滚该文件已落盘内容，不影响同批其它文件。
- 返回 `{ "uploaded": [StandardDocOut...], "failed": [{name, reason}...] }`，HTTP 200。

### 出参 `StandardDocOut`
`{ id, doc_code, title, file_name, size_bytes, mime_type, created_at }`（`file_name/size_bytes/mime_type` 来自关联 `file_object`）。

### 错误
- 下载/删除：`standard_doc` 不存在或磁盘文件缺失 → 404。

## 7. 文件存储（本地磁盘）

- `FileStorage(base_dir)`：`save(prefix, original_name, stream, max_bytes) -> StoredBlob(object_key, size_bytes, sha256)`；`object_key = f"{prefix}/{uuid4hex}{ext}"`，落盘 `base_dir/object_key`；流式分块写、累加 sha256+size，超限抛 `FileTooLargeError` 并删除残留。`delete(object_key)` 幂等。
- 存储根 `SMART_STORAGE_DIR`，默认 `./storage`（相对后端 CWD）。`backend/storage/` 入 `.gitignore`。

## 8. 配置（`backend/.env.example`）
```
SMART_DATABASE_URL=mysql+pymysql://root:root@localhost:3306/smart
SMART_STORAGE_DIR=./storage
SMART_MAX_UPLOAD_MB=50
SMART_CORS_ORIGINS=http://localhost:3200
```
- 依赖追加 `sqlmodel`（连同 fastapi/uvicorn/sqlalchemy/pymysql/python-multipart/pydantic-settings/pytest/httpx）。
