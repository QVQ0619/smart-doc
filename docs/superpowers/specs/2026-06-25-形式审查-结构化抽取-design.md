# 形式审查 · 子项目 B：结构化抽取 — 设计文档

> 日期：2026-06-25　状态：设计已确认（待用户复审 spec）
> 所属：「立项审查 AI 辅助系统」之**形式审查**能力（系统三环节的第二环：规则准备✅ → **形式审查⏳** → 专家审查⏳）
> 前置：子项目 A（申请材料上传+识别）✅完成（HEAD df48185），`material_file` + `parse_segment` 已可落库。

## 1. 背景与定位

形式审查整条流水线按**水平阶段**拆为 A→B→C→D，逐个跑通：

- A 申请材料上传 + 识别 ✅（`material_file` + 占位父行 + `parse_segment`）
- **B 结构化抽取（本文）**：把识别出的 `parse_segment` 抽成结构化审查对象，入 5 张表
- C 形式审查执行（按配置包逐条 hard 规则判定 → `round_check` + `finding_evidence`）
- D 审查报告（汇总 `review_report`）

B 的产物（成员/经费/附件/标量字段）正是 C 阶段 hard 规则（资格符合性、经费合规、材料齐全性、字段规范性）的核对对象。

**驱动架构**：延续 A 与规则准备侧——**Blade agent + skill + 幂等写端点**（cloudflared 隧道 + X-API-Key 写鉴权 + python shim）。抽取的「读段落→判断成结构化行」是 agent（LLM）的活；后端只做校验 + 幂等落库。

## 2. 本期目标与验收

**目标**：识别完成后，用户在 Blade 聊天显式说"结构化抽取这份材料" → agent 经新 skill `local/extract-material-structure` 读该审查包全部 `parse_segment` → LLM 抽成五类结构化 JSON（每行带 `source_segment_id` 溯源）→ `POST /api/packages/{id}/extract` 幂等入库；前端「审查文档库」展开审查包时能看到抽取出的结构化数据。

**验收标准**：
1. `POST /api/packages/{id}/extract` 传五类结构化 payload → 写入 `package_member`/`package_coop_unit`/`budget_item`/`package_attachment`/`extracted_field`，返回各表条数。
2. 对同一包**重跑** extract → 旧行被替换（幂等，不累积重复）。
3. 传了 `project_name` → `declared_project.project_name` 被回填（轻升格）。
4. `GET /api/packages/{id}/structured` 聚合返回五类数据；`GET /api/packages/{id}/segments` 聚合返回全包段落。
5. 前端「审查文档库」：展开审查包 → 除材料/段落外，能看到结构化只读视图（成员/合作单位/预算/附件/标量字段）。
6. 后端全量测试绿；前端 vitest 绿 + `npm run build` 零错误。

**非目标（YAGNI，留给 C/D 或更后）**：hard 规则判定与四要素；规范实体解析（`research_person`/`research_unit` 去重匹配）；预算归总层级（`budget_item` 自引用）；按 `required_material_item` 的附件齐全性核对；form_field 按 project_type 差异化；多轮次（`round_no` 恒为 1）。

## 3. Seed 扩展：form_template / form_field

`extracted_field.form_field_id` 强制 FK → `form_field` → `form_template`，而 schema 零 seed，故须幂等自建。复用 A 的 `ensure_default_master_data` 产出的默认 `project_type`/`review_stage`。

新增幂等帮助 **`ensure_default_form_template(db) -> form_template_id`**：
- `form_template`：`(project_type_id=默认, stage_id=默认, version="__DEFAULT__")` get-or-create（UNIQUE `uq_ft` 去重），name="默认申报书模板"。
- `form_field`：按固定默认集 get-or-create（UNIQUE `uq_ff(template_id, field_code)`）。默认字段集（7 个）：

  | field_code | field_name | logic_type | seq |
  |---|---|---|---|
  | project_name | 项目名称 | text | 1 |
  | project_category | 项目类别 | enum | 2 |
  | applicant_name | 申请人 | text | 3 |
  | applicant_unit | 依托单位 | text | 4 |
  | total_budget | 经费总额(万元) | number | 5 |
  | research_period | 研究周期 | text | 6 |
  | secrecy_level | 密级 | enum | 7 |

  `is_required` 全 0（本期不强制）；`logic_type` 无 schema CHECK，取上述描述值即可。

**`required_material_item` 不 seed**：`package_attachment.required_item_id` 留 NULL，附件类型靠 `attachment_type` 九枚举承载；齐全性核对留 C 阶段。

## 4. 后端端点（`routers/`，沿用 A 风格）

### 4.1 `GET /api/packages/{id}/segments`（读，无鉴权）
聚合该审查包下所有 `material_file` 的 `parse_segment`，供 agent 一次读全包。返回 `[{material_file_id, file_name, segments:[{segment_id, page_no, segment_type, content_text}]}]`。包不存在→404。

### 4.2 `POST /api/packages/{id}/extract`（写，`Depends(require_api_key)`）
- body（`MaterialExtractPayload`）：
  ```
  {
    project_name?: str,
    members:    [{member_role, name, title?, birth_date?, unit_name?, source_segment_id?}],
    coop_units: [{coop_type, unit_name, task_desc?, applied_fund?, source_segment_id?}],
    budget_items:[{category, item_name, amount, source_segment_id?}],
    attachments:[{attachment_type, is_present, source_segment_id?}],
    fields:     [{field_code, field_value?, extraction_status?, source_segment_id?}]
  }
  ```
- 逻辑：
  1. 校验 `application_package` 存在（否则 404）。
  2. `ensure_default_form_template(db)`（为 `fields` 解析 form_field_id）。
  3. 校验所有非空 `source_segment_id` 属于本包的 material_file（否则 422）。
  4. **幂等**：先删本包 `round_no=1` 的 `package_member`/`package_coop_unit`/`budget_item`/`package_attachment`/`extracted_field` 旧行，再按 payload 插入（`budget_item` 全 `is_subitem=0`/`parent_item_id=NULL`；`extracted_field` 经 `field_code`→form_field_id 映射，未知 field_code 跳过并计入 skipped；`field_code_snapshot=field_code`）。
  5. **轻升格**：若 `project_name` 非空，回填 `declared_project.project_name`。
- 校验枚举落 schema CHECK 白名单（见 §6），违例→422。
- 返回 `MaterialExtractResult`：`{members, coop_units, budget_items, attachments, fields}` 各写入条数 + `skipped_fields`。

### 4.3 `GET /api/packages/{id}/structured`（读，无鉴权）
返回五类结构化数据（含各行 `source_segment_id`），供前端 Tab 与 shim 回读校验。包不存在→404。

## 5. 落库逻辑（5 表，MVP 取直）

- `package_member`：`member_role∈{applicant,participant}`；存 `name`/`title`/`unit_name` 字符串，`birth_date` 可空；`person_id` 留 NULL（不解析规范实体）。
- `package_coop_unit`：`coop_type∈{联合承研,合作单位}`；`unit_id` NULL；`task_desc`/`applied_fund` 可空。
- `budget_item`：`category∈{设备费,业务费,劳务费,间接费,管理费}`；`amount` 必填；**MVP 扁平**（`is_subitem=0`、`parent_item_id=NULL`）。
- `package_attachment`：`attachment_type∈{推荐信,导师同意函,知情同意书,伦理证明,聘任合同,标准初稿,技术成熟度报告,社科结项证书,其他}`；`is_present` bool；`required_item_id`/`material_file_id` 可空。
- `extracted_field`：UNIQUE `(package_id, round_no, form_field_id)`；`field_value` 可空；`extraction_status∈{ok,missing,uncertain}`（缺省 ok）；`source_segment_id` 可空。

所有表 `package_id` + `round_no=1` + `source_segment_id`（溯源回 `parse_segment`）。

## 6. 错误处理与枚举校验

- 包不存在 → 404；`source_segment_id` 不属于本包 → 422；枚举值越界（member_role/coop_type/category/attachment_type/extraction_status）→ 422，错误信息点名字段。
- 无 X-API-Key → 401（写端点鉴权依赖）。
- 空抽取（识别未完成 / 无可抽内容）：payload 各数组为空 → 正常返回各 0 条，不报错。
- 未知 `field_code` → 跳过该字段并计入 `skipped_fields`（不阻断整体）。
- 前端：结构化列表 isError / 空态分别处理。

## 10. 复用与技术债

**复用**：A 的 `ensure_default_master_data`/占位父行、`materials.py`、`parse_segment` 读取、写端点 X-API-Key、前端 MaterialLibrary 只读展开模式、skill 目录布局与 shim 模式。

**技术债 / 后续**：
- 不做规范实体解析：`package_member.person_id`/`package_coop_unit.unit_id`/`declared_project.applicant_person_id` 仍为 NULL 或占位；将来做主数据管理时升格匹配。
- `budget_item` 扁平，无归总层级（自引用 `parent_item_id`）；如 C 阶段经费规则需分级再补。
- `required_material_item` 未 seed → 附件齐全性「应交 vs 实交」核对留 C 阶段；本期 `package_attachment` 仅记录实交项。
- `form_field` 为固定默认集，未按 project_type/stage 差异化模板。
- 轻升格仅回填 `project_name`；申请人/单位字段仍取占位值，C/D 若依赖须再升格。
