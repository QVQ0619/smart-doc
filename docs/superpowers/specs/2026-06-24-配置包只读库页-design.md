# 配置包(只读库页) 设计文档

> 日期：2026-06-24
> 范围：立项审查 AI 系统 · 资源库 ·「配置包」页(`lib-packs`)
> 状态：设计已与用户逐节确认，待落实现计划

## 1. 背景与目标

「规则库」页已能上传规则文件(`standard_doc`)、识别、抽取并展示其审查规则
(`review_rule` + 版本 + 出处)。现需要在并列的「配置包」页(导航 `lib-packs`，
当前为占位空页)让用户**按规则文件浏览审查配置包**：

- 每个规则文件(政策/标准文档)对应**一个**配置包；
- 包内规则 = 该文档抽出的**全部**审查规则，**自动**确定；
- 用户可**查看**包内有哪些规则，但**不能修改任何内容**(纯只读)；
- 「选择配置包后只按此包审查」是后续审查模块的消费方，**本期不做**(审查流程页
  `review-new` / `review-tasks` / `review-report` 与审查引擎均尚未实现)。

### 本期范围(明确边界)
- ✅ 做：`GET /config-packages` 只读聚合接口 + 「配置包」库页(列表 + 只读详情)。
- ❌ 不做：创建/编辑/删除配置包；勾选规则子集；选包并绑定到审查；按包跑审查。

## 2. 选定方案：纯派生(Approach A)

配置包是**现有规则的只读投影**，**不落库、不建表、不做数据迁移、无写路径**。
一个 `standard_doc`(含 ≥1 条有效规则)= 一个配置包，包内规则经现有归属链反查得到。

### 为何不物化(否决 Approach B)
schema 已预留 `config_package` / `config_rule_version` 两张空表，但它们按
「项目类型 + 阶段」建模(`project_type_id` / `stage_id` 均 NOT NULL，且**无**
`standard_doc_id` 列)，与「每个文档一个包」不匹配。本期只读浏览物化它们需要塞占位
值或加列迁移，并维护「规则变更→快照同步」，属过度工程(YAGNI)。物化推迟到将来真正
做「选包→按包跑审查」时再做，届时一并解决「按文档 vs 按项目类型+阶段」的模型落差。
此「未来可物化」仅作备注，本期不写任何预留代码(否决 Approach C 的预留口子)。

## 3. 架构与数据流

```
配置包页
  │  ① 进页面：GET /api/config-packages
  ▼
后端现场 join：standard_doc ──(review_rule → review_rule_version
                 → review_rule_clause → regulation_clause.standard_doc_id 反查归属)──▶ review_rule
  │  GROUP BY 文档：算 rule_count、聚合去重 dimensions
  ▼
返回包列表 [{doc_id, doc_code, title, version, rule_count, dimensions[]}]
  │  ② 展开某个包：GET /api/standard-docs/{doc_id}/rules  ← 复用现成接口
  ▼
只读列出该包内规则(名称/维度/判定/处置/绑定类/出处)
```

- 包身份 = `doc_id`(因与文档 1:1)。
- 规则归属沿用 `list_rules` 的同一条 join 链，故**配置包页与规则库页展开看到的规则一致**。

## 4. 后端设计

### 4.1 新增只读接口 `GET /api/config-packages`
- 鉴权：无(与现有 `GET /standard-docs/.../rules`、`/clauses`、`/segments` 一致；
  写端点才走 `require_api_key`)。
- 语义：**只含有效规则的文档**才作为一个包返回；无任何有效规则的文档**不出现**
  (空包对配置包无意义；这些文档在规则库页仍可见)。
- 实现：在 `list_rules` 同款 join
  (`standard_doc → review_rule[is_active] → review_rule_version[current]
   → review_rule_clause → regulation_clause`)上 `GROUP BY standard_doc`，
  统计 `rule_count`、聚合去重 `dimensions`(维度名)。
- 排序：按 `standard_doc.id`(与规则库列表口径一致)。
- 放置：`backend/app/routers/clauses.py`(规则/条款同域)或新建
  `routers/config_packages.py`；实现计划阶段定，倾向**新建 `config_packages.py`**
  保持单一职责。聚合查询逻辑放 `structuring.py` 或就近的查询函数中，路由只编排。

### 4.2 新增 schema `ConfigPackageOut`
```
doc_id: int
doc_code: str
title: str
version: str
rule_count: int
dimensions: list[str]   # 去重后的维度名
```

### 4.3 详情不新增
展开看包内规则直接复用现有 `GET /standard-docs/{doc_id}/rules`(已返回完整结构化
规则 + 出处)，后端零改动。

## 5. 前端设计

### 5.1 页面挂载
- `src/pages/library/ConfigPacksPage.tsx`：把占位 `EmptyState` 换成渲染新组件
  `ConfigPackageLibrary`，外壳 `PageScaffold title="配置包"` 不变(同构于
  `RuleLibraryPage` 包 `StandardDocLibrary`)。
- 新增 API 客户端 `src/api/configPackages.ts`：`listConfigPackages()`；
  包内规则**复用** `src/api/standardDocs.ts` 的 `listRules(docId)`。

### 5.2 列表(主表格)
`@tanstack/react-query` 拉 `listConfigPackages()`，antd 表格列：

| 列 | 内容 |
|---|---|
| 配置包名称 | `title` |
| 关联文件编码 | `doc_code` |
| 版本 | `version` |
| 规则数 | `rule_count` |
| 覆盖维度 | `dimensions[]` → 若干 antd `Tag` |

- **整页只读**：无上传/新建/编辑/删除按钮。
- 空状态：无任何包时显示 `EmptyState`，文案如「暂无配置包，请先在规则库中为规则
  文件抽取审查规则」。

### 5.3 只读详情(展开行)
点展开图标 → 懒加载 `listRules(doc_id)`，展开区**只读**列出每条规则：
- 规则名称、维度、判定类型(hard/verify/soft)、处置(reject/fix/review)、
  绑定类(common/parameterized/specific)；
- 出处：`第X页第Y段`(来自 `page_no` / `locator`，与规则库页一致)。
- 展开区**不含**任何编辑/删除/Tab 切换，比规则库页展开更精简。

### 5.4 实现取舍
- **不从 `StandardDocLibrary` 抽公共组件**：该组件体量大、耦合编辑/删除/弹窗，硬抽
  风险大于收益。新组件按已知 `RuleOut` 形状独立写简洁只读渲染。日后若需统一再做
  针对性提取。
- 加载/失败：列表与展开各自有 loading 占位；`listRules` 失败仅在该展开区内提示，
  不影响整页。

## 6. 错误处理

- 后端：无包返回 `[]`(200)；详情复用端点既有行为不变(文档不存在/停用 → 404，
  无规则 → `[]`)。只读聚合查询不引入新失败模式。
- 前端：列表 query `isLoading`/`isError` 分别给 loading 与「配置包加载失败」可重试；
  展开懒加载失败仅在该展开区提示；空列表渲染 `EmptyState`。

## 8. 已知技术债(沿用，不在本期解决)

`list_rules` 假定规则与文档 **1:1**；若某规则同时引用多个文档的条款(N:M)，反查会
出现重复行——这是规则库页**既有**的同一问题，本期配置包页与其表现一致，留待子项目4
统一修正。

## 9. 不做项(YAGNI)

- 不创建/编辑/删除配置包；不勾选规则子集；不跨文档拼包。
- 不物化 `config_package` / `config_rule_version`。
- 不实现选包绑定审查、不实现按包跑审查。
