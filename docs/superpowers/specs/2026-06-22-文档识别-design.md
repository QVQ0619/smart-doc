# 文档识别（子项目1）设计

> 立项审查 AI 系统「文档识别 + 规则抽取」功能的**第一个子项目**。
> 后续子项目2「规则抽取」依赖本子项目产出的 `parse_segment`。

## 目标

规则文件入库成功后，后端读取自身磁盘上的文件，把它解析成结构化的 `parse_segment`
行（带页/段出处），供后续规则抽取消费；识别失败可由前端「重新识别」按钮重跑。
识别全过程在后端同步完成，agent 仅在对话中回报识别摘要。

## 背景与既有事实

- **文件已落后端磁盘**：规则文件经 `POST /api/standard-docs` 上传后，二进制存于
  `storage/standard_doc/<uuid>.<ext>`，元数据在 `file_object`，业务行在 `standard_doc`
  （`file_id` 外键指向 `file_object`）。因此识别这步**后端读自己磁盘即可，agent/sandbox
  完全不碰文件字节**。
- **DB 已为本功能建表**（`docs/file/审查系统-schema_mysql.sql`，56 表）：
  - `standard_doc` 已带 `recognition_status`（默认 `pending`）与 `recognition_error`。
  - `parse_segment` 同时挂 `standard_doc_id` 与 `material_file_id`（均可空）——识别产物
    两边共用，申请材料审查阶段复用同一张表。字段：`page_no`、`locator`(JSON)、
    `segment_type`、`content_text`、`ocr_confidence`。
  - `regulation_clause.source_segment_id` 外键指向 `parse_segment`——子项目2 抽取出的
    规则靠它回指出处。本子项目不写 `regulation_clause`。
- **既有 skill 模式**：`SKILL.md`（教 agent 何时/怎么做）+ 纯标准库 python shim → POST
  后端 → 后端写库；落库逻辑在后端，铁律要求只走 shim、不绕过、不谎报。
- **测试基线**：后端测试连真实 MySQL `smart_test`（由 schema SQL 建库）+ FastAPI
  `TestClient`（见 `backend/tests/conftest.py`）。

## 范围

### 本子项目做

1. 后端识别核心 `recognize_standard_doc` + PDF/Word 解析器。
2. upload 成功后**自动**调用识别（best-effort，失败不影响上传）。
3. 重跑接口 `POST /api/standard-docs/{id}/recognize`。
4. 列表接口附带 `recognition_status`；上传响应附带识别摘要。
5. `save-rule-doc` shim + SKILL.md 小改：把识别摘要一并回报（交付「对话摘要」）。
6. 前端规则库：行内显示识别状态 + 「重新识别」按钮。

### 本子项目不做（明确推迟）

- **规则抽取**（写 `regulation_clause` / `review_rule*`）——子项目2。
- **skill① `recognize-document`（对话式重跑/材料场景识别）**——推迟到子项目2 或材料
  审查阶段真正需要对话识别时再做；本子项目重跑用前端按钮。
- **OCR / 扫描件**——抽不到文本即判失败并明确报错，不引入 OCR 依赖（YAGNI）。
- **前端段落级展示页**（逐段列出 `parse_segment`）——识别展示仅「对话摘要 + 落库」。
- **老式 `.doc` 二进制**——不支持，提示转存 `.docx`。

## 架构与数据流

```
POST /api/standard-docs            （已有：存文件 + 建 standard_doc）
        │  每个 doc 入库成功后，同一请求内 best-effort 调用 ↓
        ▼
recognize_standard_doc(db, storage, doc_id) -> RecognizeResult   ← 可复用核心
        │  读 file_object → 按扩展名分派 parse_pdf / parse_docx
        │  → 先删该 doc 旧 parse_segment（幂等）→ 批量写新 segment
        ▼
parse_segment 多行（standard_doc_id=本文档, material_file_id=NULL）
        │
        ▼
更新 standard_doc.recognition_status = done|failed (+ recognition_error)

重跑：POST /api/standard-docs/{id}/recognize → 同一个 recognize_standard_doc
```

**识别同步执行**（电子版文档解析很快）。识别失败**只置 `recognition_status=failed`
+ `recognition_error`，绝不让上传或重跑接口返回 5xx**（文件/业务行已存在）。

## 组件

### 后端 `backend/app/recognition.py`（新建）

```python
@dataclass
class SegmentDraft:
    page_no: int | None
    locator: dict          # 见下表
    segment_type: str      # heading | paragraph | table
    content_text: str

def parse_pdf(path: Path) -> list[SegmentDraft]: ...   # pdfplumber
def parse_docx(path: Path) -> list[SegmentDraft]: ...  # python-docx

def recognize_standard_doc(db: Session, storage: FileStorage, doc_id: int) -> RecognizeResult:
    """幂等：删该 standard_doc 旧 parse_segment → 重建 → 置 recognition_status。
    解析/格式异常一律捕获，置 failed + recognition_error，函数本身不抛业务异常。"""
```

- **`parse_pdf`**：pdfplumber 逐页。`extract_text()` 按空行切成文本块，每块一个
  `paragraph` 段（`page_no=页码(1基)`，`locator={"page":页码,"block_index":块序}`）；
  `extract_tables()` 每张表一个 `table` 段（`content_text`=行用 ` | ` 连、行间换行；
  `locator={"page":页码,"table_index":序}`）。空块跳过。`page_count`=PDF 总页数。
- **`parse_docx`**：python-docx。`document.paragraphs` 按 `style.name` 判断：以
  `Heading` 开头 → `heading`（`locator={"para_index":i,"level":级别}`），否则
  `paragraph`（`locator={"para_index":i}`）；空文本段跳过。`document.tables` 每张表一个
  `table` 段（`locator={"table_index":j}`）。全部 `page_no=NULL`（docx 无页码）。
  `page_count=NULL`。已知限制：python-docx 段落与表格分列，二者相对顺序不保留，
  本版分别用 `para_index`/`table_index` 记录，不强求全局顺序。
- `content_text` 截断上限沿用合理值（如 64KB/段，超长截断），`ocr_confidence=NULL`。

### parse_segment 落库形状

| 来源 | segment_type | page_no | locator |
|---|---|---|---|
| PDF 段落 | `paragraph` | 页码(1基) | `{page, block_index}` |
| PDF 表格 | `table` | 页码(1基) | `{page, table_index}` |
| docx 标题(由 style) | `heading` | NULL | `{para_index, level}` |
| docx 段落 | `paragraph` | NULL | `{para_index}` |
| docx 表格 | `table` | NULL | `{table_index}` |

> 注：`heading` 仅 docx 能由 style 免费拿到；PDF 不强行猜 heading，章节/条款语义留给
> 子项目2 的抽取阶段（LLM）判断。识别阶段保持纯机械、可复现。

### 后端 `backend/app/routers/standard_docs.py`（修改）

- `upload_standard_docs`：每个 doc `db.commit()` 成功后，`try: recognize_standard_doc(...)`，
  `except` 仅记录（识别函数内部已自我兜底），把识别摘要并入该项的 `StandardDocOut`。
- 新增 `POST /standard-docs/{id}/recognize` → 调 `recognize_standard_doc` → 返回
  `RecognizeResult`；`doc` 不存在/已软删 → 404。
- `list_standard_docs`：返回项带 `recognition_status`（不带 count，保持查询轻量）。

### 后端 `backend/app/schemas.py`（修改）

```python
class StandardDocOut(BaseModel):
    id: int
    doc_code: str
    title: str
    file_name: str
    size_bytes: Optional[int]
    mime_type: Optional[str]
    created_at: Optional[datetime]

    segment_count: Optional[int] = None     # 列表不填；上传响应填
    page_count: Optional[int] = None

    doc_id: int
    doc_code: str
    recognition_status: str                 # done | failed
    segment_count: int
    page_count: Optional[int] = None
    error: Optional[str] = None
```

### 后端依赖 `backend/requirements.txt`（修改）

新增 `pdfplumber`、`python-docx`。

### Agent shim 与 skill（修改，小改）

- `backend/agent_shim/smart_doc_add.py`：解析上传响应时，若 `uploaded` 项含
  `recognition_status`，stdout 额外打印 `recognition=<status> segments=<n>`（向后兼容，
  字段缺失则不打印）。
- `blade/skills/save-rule-doc/SKILL.md`：在「解读结果」补一句——入库成功后若响应带识别

  `recognition=failed`，照实说「已入库但识别失败：<原因>，可在规则库页点重新识别」。

### 前端规则库（修改）

- `StandardDocLibrary` 组件：列表渲染每行显示 `recognition_status` 徽标

- 每行加「重新识别」按钮 → 调 `POST /api/standard-docs/{id}/recognize` → 成功后刷新列表
  / 用 sonner 提示「已识别 N 段」或失败原因。
- API 客户端加 `recognizeStandardDoc(id)`。

## 错误处理

识别失败一律：`recognition_status=failed`，`recognition_error` 写明确中文原因，
**接口仍返回 2xx**（上传/重跑都不因识别失败而 5xx）。

| 情形 | recognition_error |
|---|---|
| 不支持的格式（.doc/图片等） | `不支持的格式 <ext>，请转存为 .docx 或 PDF` |
| PDF 抽不到任何文本（疑似扫描件） | `未从 PDF 抽取到文本，可能是扫描件（暂不支持 OCR）` |
| 解析库抛异常 | `识别失败：<异常信息截断 300 字>` |
| 文件在磁盘缺失 | `源文件缺失，无法识别` |

## 测试策略（沿用 MySQL `smart_test` + TestClient）

- **`parse_docx` 单测**：测试内用 python-docx 现造 docx（含 Heading 段、普通段、一张表）
  → 断言 segment 列表的 `segment_type`/`locator`/`content_text`/`page_no=None`。
- **`parse_pdf` 单测**：提交一个小固定 fixture PDF（`backend/tests/fixtures/`，已知 2 页
  文本 + 一张表）→ 断言 `page_no`、`block_index` 顺序、表格段。
- **`recognize_standard_doc` 集成测**：构造 file_object+standard_doc+磁盘文件 → 调用 →
  断言 `parse_segment` 行数、`recognition_status=done`；**幂等**：再调一次断言行数不累加
  （替换）；不支持格式 → `failed` + 非空 `recognition_error`，且不抛异常。
- **接口测**：`POST /{id}/recognize` 返回 `RecognizeResult`（含 segment_count）；不存在
  id → 404；`POST /standard-docs` 上传 docx 后响应项 `recognition_status=done` 且
  `segment_count>0`；`GET /standard-docs` 项含 `recognition_status`。
- **shim 单测**：`smart_doc_add` 解析含 `recognition_status` 的响应时多打印一行；
  字段缺失时行为不变（向后兼容）。

## 验收标准

1. 上传一个电子版 PDF 规则文件，`standard_doc.recognition_status` 变 `done`，
   `parse_segment` 出现该文档的多行（带正确 `page_no`/`locator`/`content_text`），
   上传响应 `segment_count>0`。
2. 上传一个 `.docx`（含标题/段落/表格），识别出 `heading`/`paragraph`/`table` 段，
   `page_no` 均为 NULL。
3. 上传扫描件 PDF（无文字层）→ `recognition_status=failed`，`recognition_error` 明确，
   上传接口仍 2xx、文件与 `standard_doc` 仍在。
4. 规则库页每行显示识别状态徽标；点「重新识别」可重跑，结果幂等（segment 不累加）。
5. 对话中入库成功后，agent 回报里含识别摘要（成功段数 / 失败原因）。
6. 后端全量测试在 `smart_test` 上通过；前端相关测试通过、`npm run build` 干净。
