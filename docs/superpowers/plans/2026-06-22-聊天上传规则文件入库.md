# 聊天上传规则文件入库（Blade agent → 规则库）Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 让用户在 Blade 聊天里附带上传规则文件并发指令，由 agent 调一个薄 shim 把原文件存入规则库（复用现有后端接口），规则库页轮询自动出现新行。

**Architecture:** 方案 A —— 后端零改动，复用 `POST /api/standard-docs`。新增纯 stdlib 的 `smart-doc-add` shim（部署到 agent 容器）把 sandbox 文件 multipart POST 到后端；新增 `save_rule_doc` skill 指挥 agent；前端规则库页 `useQuery` 加 10s 轮询。

**Tech Stack:** Python 3.13（标准库 urllib/http.server）、pytest；React 19 + `@tanstack/react-query` + vitest；Blade `@blade-hq/agent-kit@0.5.11`（聊天附件原生）。

## Global Constraints

- **后端零改动**：唯一依赖是已存在的 `POST /api/standard-docs`（multipart，字段名 **`files`**），返回 `{"uploaded":[{id,doc_code,title,file_name,size_bytes,mime_type,created_at}],"failed":[{name,reason}]}`，整体 200。
- shim **纯标准库**，无第三方依赖（urllib + 手搓 multipart）。
- shim 退出码语义固定：`0`全部成功 / `1`用法错误 / `2`文件不存在 / `3`网络错误 / `4`后端非2xx / `5`有failed项 / `6`缺 `SMART_DOC_API`。multipart 字段名必须是 `files`。
- 后端命令用 `backend/.venv/Scripts/python`，CWD=`E:\shixi\smart-doc\backend`；前端命令 CWD=仓库根。
- 现有测试不回归：后端 15、前端 26。
- 规则库页轮询间隔 **10000ms**。
- 本期只做规则文件原文件入库；不做内容提取、不做完整 CLI 框架、不做前端 tool-call 拦截。
- 不要 `git add -A`：仓库存在未跟踪的 `docs/file/`、`docs/参考文件/` 参考目录，逐文件 `git add`。

```python
#!/usr/bin/env python3
"""smart-doc-add: 把 sandbox 里的规则文件上传到 smart-doc 后端入规则库.

纯标准库; 供 Blade agent 容器挂载为 PATH 上的可执行 `smart-doc-add` 使用.
用法: smart-doc-add <path> [<path> ...]
环境: SMART_DOC_API(必填, 后端 base URL), SMART_DOC_TIMEOUT(默认120秒)
退出码: 0全部成功 1用法错误 2文件不存在 3网络错误 4后端非2xx 5有failed项 6缺SMART_DOC_API
"""
from __future__ import annotations

import json
import mimetypes
import os
import sys
import urllib.error
import urllib.request
import uuid
from pathlib import Path

FIELD = "files"


def build_multipart(paths):
    """构造 multipart/form-data, 每个文件放在字段 'files'. 返回 (body: bytes, content_type: str)."""
    boundary = uuid.uuid4().hex
    crlf = b"\r\n"
    parts = []
    for p in paths:
        path = Path(p)
        data = path.read_bytes()
        ctype = mimetypes.guess_type(path.name)[0] or "application/octet-stream"
        parts.append(b"--" + boundary.encode() + crlf)
        parts.append(
            f'Content-Disposition: form-data; name="{FIELD}"; filename="{path.name}"'.encode("utf-8")
            + crlf
        )
        parts.append(f"Content-Type: {ctype}".encode("utf-8") + crlf + crlf)
        parts.append(data + crlf)
    parts.append(b"--" + boundary.encode() + b"--" + crlf)
    return b"".join(parts), f"multipart/form-data; boundary={boundary}"


def upload(api_base, paths, timeout):
    """POST 到 {api_base}/api/standard-docs, 返回解析后的 dict.
    非2xx 抛 urllib.error.HTTPError; 连接失败抛 urllib.error.URLError."""
    body, ctype = build_multipart(paths)
    url = api_base.rstrip("/") + "/api/standard-docs"
    req = urllib.request.Request(url, data=body, method="POST", headers={"Content-Type": ctype})
    with urllib.request.urlopen(req, timeout=timeout) as resp:
        return json.loads(resp.read().decode("utf-8"))


def main(argv):
    paths = argv[1:]
    if not paths:
        sys.stderr.write("用法: smart-doc-add <path> [<path> ...]\n")
        return 1
    api_base = os.environ.get("SMART_DOC_API")
    if not api_base:
        sys.stderr.write("[smart-doc-add] 未设置环境变量 SMART_DOC_API\n")
        return 6
    timeout = int(os.environ.get("SMART_DOC_TIMEOUT", "120"))

    for p in paths:
        if not Path(p).is_file():
            sys.stderr.write(f"[smart-doc-add] 文件不存在: {p}\n")
            return 2

    try:
        result = upload(api_base, paths, timeout)
    except urllib.error.HTTPError as e:  # 必须在 URLError 之前(HTTPError 是其子类)
        detail = e.read().decode("utf-8", "ignore")[:300]
        sys.stderr.write(f"[smart-doc-add] 后端返回 {e.code}: {detail}\n")
        return 4
    except urllib.error.URLError as e:
        sys.stderr.write(f"[smart-doc-add] 网络错误: {e.reason}\n")
        return 3

    for doc in result.get("uploaded", []):
        sys.stdout.write(
            f"doc_code={doc.get('doc_code')} title={doc.get('title')} file={doc.get('file_name')}\n"
        )
    failed = result.get("failed", [])
    for f in failed:
        sys.stderr.write(f"FAILED {f.get('name')}: {f.get('reason')}\n")
    return 5 if failed else 0


if __name__ == "__main__":
    sys.exit(main(sys.argv))
```

# save_rule_doc — 保存规则文件到规则库

## 何时触发
用户在聊天中**已附带上传一个文件**（文件已在当前 session 的 sandbox 中），并表达把它存为规则文件的意图，例如：
- "把这个存为规则文件"
- "加入规则库 / 保存到规则库"
- "这是一份规则文档，存一下"

## 前置判断
1. 确认 sandbox 中确有用户刚上传的文件，拿到它的绝对路径。
2. 判断它是否为**规则类文档**（政策、指南、规范、办法等）。
   - 若明显不是规则文档（随手的图片、无关表格等），**先说明并询问用户是否仍要保存**，不要自作主张入库。
   - 若用户意图不明确，先澄清再动作。

## 执行
对确认要保存的文件，运行（每个文件路径作为参数，可多个）：

```bash
smart-doc-add "<sandbox 中的文件绝对路径>"
```

`smart-doc-add` 会把文件上传到规则库后端并入库。

## 解读结果（按退出码）
| 退出码 | 含义 | 你应如何回报 |
|---|---|---|
| 0 | 全部成功 | 报告每个文件的标题与编号，例："已存入规则库：政策A（SD-abc123）" |
| 1 | 用法错误（没给文件） | 说明没有可保存的文件 |
| 2 | 文件不存在/不可读 | 说明没找到文件，请用户确认是否已上传 |
| 3 | 连不上规则库后端 | 说明后端暂时不可达，建议稍后重试 |
| 4 | 后端返回错误 | 说明入库失败（附 stderr 原因），不要声称成功 |
| 5 | 部分/全部文件失败 | 逐个报告：成功的报标题+编号，失败的报 stderr 里的原因（如超过大小上限） |
| 6 | 后端地址未配置 | 这是部署问题，提示需要配置 SMART_DOC_API |

## 铁律
- **只有退出码 0 且 stdout 出现 `doc_code=...` 才能说"已保存成功"。** 其他情况一律如实说明失败原因，绝不谎报。
- 不要凭空编造 doc_code 或标题——只转述 `smart-doc-add` 的实际输出。
